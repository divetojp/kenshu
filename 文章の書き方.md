# 文章の構造的な書き方

## なぜ「構造」が必要か

技術的に正しいことを書いても、読み手に伝わらなければ意味がありません。  
構造とは「読み手が迷わずに読み進められる地図」のことです。

良い技術文章の三原則：

- **明確**：一文に一つの意味だけを入れます
- **簡潔**：必要なことだけを書きます
- **具体**：抽象的な説明には必ず例を添えます

---

## はじめて読む人へ

技術を学ぶとき、文章力は後回しにされがちですが、実は理解そのものを助けます。自分の言葉で説明できない概念は、まだ頭の中で整理されていないことが多いからです。


### 読む前に押さえること

- 良い文章は、読み手だけでなく書き手の思考も整理します。
- 結論・理由・例の順に書くと、技術説明はかなり読みやすくなります。
- コードを書くときも、前後の説明があるだけで学習効率が上がります。

### 読み終えたら説明できること

- PREP法を使って短い技術説明を書ける。
- 見出しだけで文書の流れが分かる状態を作れる。
- コードブロックの前後に必要な説明を添えられる。

---

## PREP 法

PREP 法は、技術説明で特に使いやすい型です。なぜなら、読み手は最初に結論を知ると、その後の理由やコード例を受け取りやすくなるからです。

技術文書では、背景を長く書いてから結論に進むと、読み手が「結局何が言いたいのか」を見失いやすくなります。最初に主張を置き、そのあとで理由と例を示すことで、初学者でも読み進めやすい文章になります。

意見や説明を伝えるときの基本構造です。

| 順序 | 英語 | 日本語 | 役割 |
|------|------|--------|------|
| 1 | Point | 結論 | 最初に「何が言いたいか」を宣言します |
| 2 | Reason | 理由 | なぜそうなのかを説明します |
| 3 | Example | 具体例 | 実例・コード・数値で裏付けます |
| 4 | Point | 結論（再） | 最初の結論を言い換えて締めます |

**悪い例：**

> インデックスは B-tree というデータ構造を使っています。B-tree はノードが複数の子を持てます。そのため検索が速いです。データベースのクエリが遅いときはインデックスを張りましょう。

悪い例は、情報自体は間違っていません。しかし、最初に結論がないため、読み手は「何のためにB-treeの説明を読んでいるのか」を途中まで判断できません。技術説明では、正確さだけでなく、読み手が迷わず受け取れる順番が重要です。

**良い例（PREP）：**

> インデックスを張るとクエリが劇的に速くなる（Point）。  
> テーブル全行を走査する代わりに、B-tree 構造で目的の行へ直接ジャンプできるからだ（Reason）。  
> 100 万行のテーブルで `WHERE email = '...'` を実行すると、インデックスなしでは全件スキャン（O(n)）、インデックスありでは O(log n) で済む（Example）。  
> クエリが遅いと感じたら、まずインデックスの有無を確認しよう（Point）。

良い例では、最初に主張を置き、その後で理由と例を出しています。初学者は「いま何を理解すればよいのか」が分かるため、B-tree や O(log n) のような用語が出てきても、読み進める目的を失いにくくなります。

## ピラミッド構造

文書全体のレベルでも「結論を先に、詳細を後に」を徹底します。

!!! info ""
            ┌─────────────┐
            │  結論・主張  │  ← 読み手が最初に見ます
            └──────┬──────┘
           ┌───────┴───────┐
           │  理由 1  理由 2  │  ← 結論を支える柱
           └───┬───────┬───┘
          ┌────┴──┐ ┌──┴────┐
          │ 詳細  │ │ 詳細  │  ← 各柱の根拠・具体例
          └───────┘ └───────┘

新聞記事が「見出し → リード → 本文」という構造なのと同じ原理です。  
読み手は途中で読むのをやめても、重要な情報を受け取れます。

教科書を書くときも、最初に全体像を示し、そのあとで個別の概念やコード例に入ると読みやすくなります。コードブロックを先に置くのではなく、「このコードはどの概念を確認するためのものか」を先に説明すると、読者はコードの目的を持って読めます。

## 見出しの使い方

見出しは「目次の代わり」であり「ナビゲーション」です。

**良い見出しの条件：**
- 具体的で、その節を読まなくても内容が推測できます
- 動詞や名詞で体言止めにします（疑問形も有効です）

**悪い例：**
!!! info ""
    ## 説明
    ## 詳細
    ## その他

この見出しでは、何について説明しているのかが分かりません。読み手は本文を開くまで内容を予測できず、あとから復習するときにも探しにくくなります。

**良い例：**
!!! info ""
    ## インデックスが効かないクエリのパターン
    ## トランザクション分離レベルの選び方
    ## デッドロックを防ぐ 3 つのルール

良い見出しは、読む前から中身の焦点が分かります。技術文書では、見出しがそのまま目次や検索の入口になります。本文を増補するときも、「この説明はどの見出しの下に置くと一番自然か」を考えると、後から読み返しやすくなります。

## 段落の作り方

1 段落 = 1 トピックです。段落の最初の文（トピック文）でその段落の主張を言い切ります。  
残りの文はトピック文を補足・説明するためだけに使います。

**確認方法：** 各段落の最初の文だけ抜き出して並べると、文書全体の要約になるはずです。

## 技術文書で使う表現パターン

### 因果を示す

!!! info ""
    〜であるため、…となります。
    〜の結果、…が起きます。
    〜しない場合、…になります。

### 対比する

!!! info ""
    〜に対して、…は…。
    〜とは異なり、…は…。

### 手順を示す

番号リストを使います。「まず／次に／最後に」は曖昧になりがちなので避けましょう。

```markdown
1. 〜します
2. 〜を確認します
3. 〜を実行します
```

手順を書くときは、読者がそのまま実行できる粒度にします。「設定する」「確認する」だけでは抽象的すぎる場合があります。必要なら、どのファイルを開くのか、どのコマンドを実行するのか、成功すると何が表示されるのかまで書きます。

### 注意・例外を示す

!!! warning ""
    ただし、〜の場合は例外で…。
    注意：〜のとき、…になります。

## コードブロックの書き方

コードを書くときは必ず言語を指定し、そのコードが「何をするコードか」を直前の文で説明します。

**悪い例：**

> コードは以下のとおりです。
>
> ```
> SELECT * FROM users WHERE id = 1;
> ```

悪い例では、コードが何を確認するためのものか、実行すると何が返るのか、なぜこの書き方でよいのかが分かりません。コードブロックだけを置くと、読者は「写せばよいのか、仕組みを理解すべきなのか」を判断できなくなります。

**良い例：**

> `id` を指定してユーザーを 1 件取得します。
>
> ```sql
> SELECT id, name, email
> FROM users
> WHERE id = 1;
> ```
>
> `SELECT *` ではなく必要なカラムだけを指定することで、不要なデータ転送を避けられます。

良い例では、コードの前に目的、コードの中に具体例、コードの後に設計上の理由があります。この3点がそろうと、コードは単なるサンプルではなく、概念を理解するための教材になります。

## 文章セルフチェックリスト

書き終えたら以下を確認します。

- [ ] 結論（何が言いたいか）が冒頭にあるか
- [ ] 各見出しを読むだけで内容が推測できるか
- [ ] 抽象的な説明に具体例が添えられているか
- [ ] 1 文が 60 字を超えていないか（超える場合は分割を検討）
- [ ] 「〜です・ます」で統一されているか
- [ ] コードブロックに言語指定（` ```sql ` など）があるか
