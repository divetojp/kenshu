# コーディングテスト対策

> [アルゴリズム・データ構造](アルゴリズム-データ構造) で理論を学んだあとに読んでください。就活の選考テストで実際に問われるパターンと、解く手順をまとめています。

## はじめて読む人へ

コーディングテストでは、限られた時間で問題を読み、適切なアルゴリズムを選び、正しく実装する力が問われます。暗記よりも、問題を型に分けて考えることが大切です。


### 読む前に押さえること

- 二分探索は、単調性がある問題で使えます。
- BFS/DFS は、グラフや状態遷移を探索するときに使います。
- 動的計画法は、部分問題の結果を再利用する考え方です。

### 読み終えたら説明できること

- 代表的な問題パターンを見分けられる。
- 計算量を考えて解法を選べる。
- 実装後に境界条件を確認できる。

## 問題を解く手順

コーディングテストでは、最初に問題文を実装へ翻訳します。入力は何か、出力は何か、制約はどれくらいか、愚直解では間に合うかを確認します。

制約を見ると、必要な計算量の目安が分かります。n が $10^5$ なら $O(n^2)$ は厳しい、n が 20 程度なら全探索やビットDPが使える、というように解法を絞れます。

どんな問題でも、コードを書く前に以下を必ず行ってください。

```
1. 入出力を確認する（何を受け取り、何を返すか）
2. 制約を確認する（n の最大値 → 計算量の上限が決まる）
3. 例を手で解いてみる（2〜3 個のサンプルで確認）
4. アルゴリズムを決める（脳内でどう解くか）
5. 実装する
6. エッジケースを確認する（n=0、全要素同じ、最大値など）
```

**制約から計算量を逆算する目安：**

| n の上限 | 許容できる計算量 | アルゴリズムの例 |
|---------|--------------|---------------|
| $n \leq 10^6$ | $O(n \log n)$ | ソート・二分探索 |
| $n \leq 10^5$ | $O(n \log n)$ 〜 $O(n)$ | BFS/DFS・ハッシュ |
| $n \leq 10^3$ | $O(n^2)$ | 2 重ループ・DP（小規模） |
| $n \leq 20$ | $O(2^n)$ | 全探索・ビット DP |

## 頻出パターン 1：二分探索

「ソート済み配列から探す」「最小値を最大化する」問題に使います。

二分探索には2種類の使い方があります。1つ目は、ソート済み配列の中から値や挿入位置を探す使い方です。2つ目は、「条件を満たす最小値」「条件を満たす最大値」のように、答えそのものを探索する使い方です。

```python
import bisect

# ソート済みリストに値を挿入する位置を O(log n) で求める
arr = [1, 3, 5, 7, 9]
pos = bisect.bisect_left(arr, 6)  # → 3（arr[3]=7 の前）

# 条件を満たす最小値を二分探索で求める（答えを探す二分探索）
def is_possible(mid: int) -> bool:
    # mid で条件を満たすか判定するロジック
    return True  # 例

lo, hi = 0, 10**9
while lo < hi:
    mid = (lo + hi) // 2
    if is_possible(mid):
        hi = mid
    else:
        lo = mid + 1
answer = lo
```

`bisect_left` は、ソート済みリストに値を入れるならどこかを返します。後半のコードでは、`is_possible(mid)` が真になる最小の `mid` を探しています。`lo` と `hi` の範囲を半分ずつ狭めるので、探索範囲が10億でも30回程度で答えに近づけます。

## 頻出パターン 2：スライディングウィンドウ

「連続する部分配列の中で条件を満たすもの」を $O(n)$ で探します。

毎回長さ `k` の部分配列を合計し直すと $O(nk)$ になります。スライディングウィンドウでは、右端に入った値を足し、左端から出た値を引くことで、合計を $O(1)$ で更新します。

```python
# 問題例：長さ k の連続部分配列の最大和
def max_sum_subarray(arr: list[int], k: int) -> int:
    window_sum = sum(arr[:k])
    max_sum = window_sum
    for i in range(k, len(arr)):
        window_sum += arr[i] - arr[i - k]  # 右を追加、左を除去
        max_sum = max(max_sum, window_sum)
    return max_sum
```

この関数では、最初の `k` 個の合計を作り、その後は窓を1つずつ右へ動かします。`arr[i]` が新しく入る値、`arr[i - k]` が窓から出る値です。連続区間の最大・最小・個数を扱う問題でよく使います。

## 頻出パターン 3：BFS（幅優先探索）テンプレート

「最短ステップ数を求める」問題に使います。

BFS は、スタートから近い順に探索します。キューに「現在地」と「そこまでの距離」を入れておくと、ゴールに到達した瞬間の距離が最短ステップ数になります。

```python
from collections import deque

def bfs(graph: dict, start, goal) -> int:
    queue = deque([(start, 0)])  # (現在地, ステップ数)
    visited = {start}
    while queue:
        node, dist = queue.popleft()
        if node == goal:
            return dist
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append((neighbor, dist + 1))
    return -1  # 到達不可
```

`visited` は、同じノードを何度も訪れないための集合です。これがないと、グラフに循環がある場合に無限ループすることがあります。迷路、SNSの友人距離、状態遷移の最短手数などで使います。

## 頻出パターン 4：DFS（深さ優先探索）テンプレート

「全探索・組み合わせ列挙・連結成分の確認」に使います。

DFS は、1つの道を行けるところまで進み、行き止まりになったら戻る探索です。再帰で書くと短くなりますが、深いグラフでは再帰上限に注意します。

```python
def dfs(graph: dict, node, visited: set):
    visited.add(node)
    for neighbor in graph[node]:
        if neighbor not in visited:
            dfs(graph, neighbor, visited)

# 再帰が深くなる場合はスタックで実装します
def dfs_iterative(graph: dict, start):
    stack = [start]
    visited = {start}
    while stack:
        node = stack.pop()
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                stack.append(neighbor)
```

再帰版は概念を理解しやすく、反復版は再帰の深さ制限を避けられます。木構造の探索、連結成分の数え上げ、依存関係のたどり方などで頻出です。

## 頻出パターン 5：動的計画法（DP）

「小さな問題の答えを使って大きな問題を解く」手法です。

DPでは、同じ計算を何度も繰り返さないように、途中結果を配列や辞書に保存します。重要なのは、`dp[i]` が何を意味するかを日本語で定義することです。定義が曖昧なまま実装すると、遷移式も曖昧になります。

```python
# 問題例：コイン問題（最小枚数で金額 amount を作る）
def coin_change(coins: list[int], amount: int) -> int:
    dp = [float("inf")] * (amount + 1)
    dp[0] = 0
    for i in range(1, amount + 1):
        for coin in coins:
            if coin <= i:
                dp[i] = min(dp[i], dp[i - coin] + 1)
    return dp[amount] if dp[amount] != float("inf") else -1
```

この例では、`dp[i]` を「金額 `i` を作るために必要な最小コイン枚数」と定義しています。`dp[i - coin] + 1` は、「金額 `i - coin` を作ったあと、coinを1枚足して `i` を作る」という意味です。

**DP を使うサイン：**
- 「最小・最大・何通り・可能かどうか」を問われる
- 「前の状態から今の状態を計算できる」

## 頻出パターン 5.5：ビット DP（n ≤ 20 の全探索）

計算量表の「n ≤ 20 → O(2^n)」のところで登場する手法です。

**考え方：** n 個の要素の「選ぶ・選ばない」を 2 進数で表すと、0〜2ⁿ−1 の整数がすべての組み合わせに対応します。例えば n=3 なら：

```
000 → 何も選ばない
001 → 要素 0 だけ選ぶ
010 → 要素 1 だけ選ぶ
011 → 要素 0 と 1 を選ぶ
100 → 要素 2 だけ選ぶ
111 → 全部選ぶ
```

n=20 なら 2²⁰ ≈ 100 万通りです。全状態をループで回しても現実的な時間で終わります。

```python
# 問題例：n 個の商品から選んで重さの合計が W 以下になる最大価値を求める
def knapsack_bitmask(weights: list[int], values: list[int], W: int) -> int:
    n = len(weights)
    best = 0
    for state in range(1 << n):          # 0 から 2^n - 1 まで全状態を列挙
        total_w = total_v = 0
        for i in range(n):
            if state & (1 << i):         # i ビット目が立っている = 要素 i を選ぶ
                total_w += weights[i]
                total_v += values[i]
        if total_w <= W:
            best = max(best, total_v)
    return best
```

`state` は、どの商品を選んだかを表すビット列です。`state & (1 << i)` が真なら、商品 `i` を選んでいます。全状態を調べるので単純ですが、`n` が20程度なら現実的に動きます。

**ビット演算の読み方：**

| 式 | 意味 |
|----|------|
| `1 << i` | 1 を i ビット左にシフト → i 番目のビットだけ 1 になります |
| `state & (1 << i)` | state の i ビット目が立っているか確認します |
| `state \| (1 << i)` | state の i ビット目を立てます |
| `1 << n` | 2ⁿ（全状態数）を表します |

> n が 20 を超えると 2²¹ ≈ 200 万、2²⁵ ≈ 3,300 万 となり TLE（時間制限超過）しやすいです。必ず制約の n ≤ 20 を確認してから使ってください。

## 頻出パターン 6：ハッシュで O(1) に変換

2 重ループを辞書（セット）で $O(n)$ に改善します。

```python
# 問題：配列に重複する要素があるか
# O(n^2) 版
def has_duplicate_slow(arr):
    for i in range(len(arr)):
        for j in range(i + 1, len(arr)):
            if arr[i] == arr[j]:
                return True
    return False

# O(n) 版：ハッシュセットを使います
def has_duplicate_fast(arr):
    seen = set()
    for x in arr:
        if x in seen:
            return True
        seen.add(x)
    return False
```

遅い版では、各要素について後ろの要素をすべて調べるため $O(n^2)$ です。速い版では、これまで見た値を `seen` に保存し、次の値がすでにあるかを $O(1)$ 平均で確認します。「過去に出たか」「ペアになる値があるか」という問題では、ハッシュを使えないか考えるのが定石です。

1. まず全探索（$O(n^2)$ でもよい）で動くコードを書きます
