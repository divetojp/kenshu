# コンパイラの仕組み

> [C 言語入門](C言語入門) でコンパイルを体験しました。このページでは「ソースコードが実行されるまで」を内部から理解します。

---

## はじめて読む人へ

コンパイラは、人間が書いたプログラムをコンピュータが実行しやすい形へ変換する仕組みです。字句解析、構文解析、最適化、コード生成という段階を通ります。


### 読む前に押さえること

- 字句解析は、文字列を意味のあるトークンに分けます。
- 構文解析は、トークンの並びが文法として正しいかを調べます。
- AST は、プログラムの構造を木として表したものです。

### 読み終えたら説明できること

- ソースコードが実行可能になるまでの流れを説明できる。
- トークン、AST、最適化の意味を理解できる。
- インタプリタとコンパイラの違いを説明できる。

---

## コンパイラとインタープリタ

プログラムを実行する方法は大きく 2 つに分かれます。

**コンパイラ（Compiler）**

ソースコード全体を先に機械語（実行ファイル）に翻訳し、その後実行します。

ソースコード → [コンパイラ] → 実行ファイル → [CPU] → 実行
例：C, C++, Go, Rust

コンパイラ方式では、実行前に時間をかけて変換する代わりに、実行時にはCPUが直接読める形式に近いコードを動かせます。C言語で `gcc main.c -o main` のように実行ファイルを作るのは、この方式です。

**インタープリタ（Interpreter）**

ソースコードを 1 行ずつ読みながら即座に実行します。

ソースコード → [インタープリタ] → 実行（同時進行）
例：Python（CPython）, Ruby, Bash

インタープリタ方式では、プログラムを書いてすぐ実行しやすい反面、実行中に読み取りや解釈のコストがかかります。Python が学習やデータ分析で使いやすいのは、コンパイル手順を強く意識せずに試せるからです。

**JIT コンパイラ（Just-In-Time）**

最初はインタープリタとして動き、よく使われるコードを実行中に機械語に変換して高速化します。

例：JavaScript（V8）, Java（JVM）, Python（PyPy）

| 方式 | 起動速度 | 実行速度 | デバッグのしやすさ |
|------|---------|---------|-----------------|
| コンパイラ | 遅い（ビルド時間が必要） | 速い | やや難しい |
| インタープリタ | 速い | 遅い | しやすい |
| JIT | 速い | 速い（温まると） | 中程度 |

## コンパイルの全体像

コンパイルは、ソースコードをいきなり機械語へ変換する単純な作業ではありません。字句解析、構文解析、意味解析、最適化、コード生成という複数の段階に分かれています。

この段階を知ると、エラーの種類を切り分けやすくなります。文字の並びが文法として読めないのか、型や名前の意味が合わないのか、実行時に問題が起きているのかで、調べる場所が変わります。

C 言語をコンパイルするときの流れです。

ソースコード (.c)
↓ 1. 字句解析（Lexing）
トークン列
↓ 2. 構文解析（Parsing）
AST（抽象構文木）
↓ 3. 意味解析（Semantic Analysis）
型チェック済み AST
↓ 4. 最適化（Optimization）
中間表現（IR）
↓ 5. コード生成（Code Generation）
アセンブリ (.s)
↓ 6. アセンブル
オブジェクトファイル (.o)
↓ 7. リンク（Link）
実行ファイル (a.out)
この流れでは、ソースコードは「文字列」から「構造」へ、さらに「機械が実行できる表現」へと段階的に変換されます。各段階で役割が違うため、エラーの出方も変わります。`SyntaxError` は構文解析付近、型エラーは意味解析付近、リンクエラーは最後の結合段階で起きる、というように見分けられます。

## 1. 字句解析（Lexing / Tokenization）

ソースコードの文字列を **トークン（意味のある最小単位）** に分割します。

```c
int x = 10 + 20;
```

↓ 字句解析後

```
[INT]  [IDENTIFIER: x]  [ASSIGN: =]  [NUMBER: 10]  [PLUS: +]  [NUMBER: 20]  [SEMICOLON: ;]
```

字句解析では、まだ「足し算の左右が何か」までは深く考えません。`int` はキーワード、`x` は識別子、`10` と `20` は数値、`+` は演算子、`;` は文の終わり、というように文字列を部品へ分けます。

字句解析器（レキサー）は **正規表現** で書かれたルールで各トークンを識別します（正規表現の重要な応用例です）。

| トークン種別 | 正規表現パターン |
|------------|----------------|
| 整数リテラル | `[0-9]+` |
| 識別子 | `[a-zA-Z_][a-zA-Z0-9_]*` |
| 演算子 | `+`, `-`, `*`, `/`, `=`, `==`, `!=` など |
たとえば `[0-9]+` は1文字以上の数字列を表します。`[a-zA-Z_][a-zA-Z0-9_]*` は、最初は英字または `_`、2文字目以降は英数字または `_` を許す識別子のルールです。プログラミング言語の「変数名として使える文字」の規則は、この段階で扱われます。

---

## 2. 構文解析（Parsing）

トークン列を **AST（Abstract Syntax Tree：抽象構文木）** に変換します。文の構造を木として表したものです。

```c
int x = 10 + 20;
```

↓ 構文解析後（AST）

```
VarDeclaration
├─ type: int
├─ name: x
└─ initializer:
    BinaryExpression(+)
    ├─ left:  NumberLiteral(10)
    └─ right: NumberLiteral(20)
```

AST は、括弧やセミコロンのような表面的な記号を取り除き、プログラムの意味の骨格を木で表したものです。`10 + 20` は、`+` という演算の左に `10`、右に `20` がある構造として表されます。

構文規則（文法）は **文脈自由文法（BNF 記法）** で定義されます。

```
expression := expression '+' term
            | term
term       := NUMBER
            | IDENTIFIER
            | '(' expression ')'
```

**構文エラーはここで検出されます：**

```c
int x = + 10;   // 意味はおかしいが構文的にはエラーになる場合も
int x = (10;    // → SyntaxError: 閉じ括弧がない
```

構文解析は「文法として並び方が正しいか」を見ます。変数が宣言済みか、型が合っているかまでは次の意味解析で扱います。自然言語でいうと、まず文の形が文法として成立しているかを確認し、その後で意味が通るかを見るイメージです。

---

## 3. 意味解析（Semantic Analysis）

AST が **意味的に正しいか** をチェックします。構文的に正しくても意味がおかしいケースを検出します。

**型チェック：**

```c
int x = "hello";   // int に文字列を代入 → 型エラー
```

**未定義変数のチェック：**

```c
int y = x + 1;   // x が宣言されていない → エラー
```

**型推論（TypeScript・Rust など）：**

```ts
const x = 10;     // x の型を number と推論
const y = x + "!";  // number + string → TypeScript はエラーを出す
```

TypeScript のコンパイラ（tsc）はここで多くのバグを事前に発見します（[TypeScript](TypeScript) 参照）。

意味解析では、ASTに対して「この名前は定義されているか」「この演算はこの型に対して許されるか」「戻り値の型は合っているか」を確認します。構文的には正しいコードでも、`int x = "hello";` のように意味的におかしいコードはここで止められます。

---

## 4. 最適化（Optimization）

プログラムの意味を変えずに、より速く・小さくなるよう変換します。

**定数畳み込み（Constant Folding）：**

```c
int x = 10 + 20;  // コンパイル時に計算
// ↓ 最適化後
int x = 30;       // 実行時の計算が不要になる
```

定数畳み込みでは、実行しなくても結果が分かる計算をコンパイル時に済ませます。`10 + 20` は入力や外部状態に依存しないため、実行時に毎回計算する必要がありません。

**デッドコード除去：**

```c
if (false) {
    doSomething();  // 絶対に実行されない → コンパイラが除去
}
```

デッドコード除去は、結果に影響しない処理を取り除く最適化です。到達不能なコードや、計算しても使われない値が対象になります。プログラムの意味を変えない範囲で、実行する命令数を減らします。

**インライン展開：**

```c
int square(int n) { return n * n; }
int result = square(5);

// ↓ インライン展開後（関数呼び出しのオーバーヘッドを除去）
int result = 5 * 5;
```

インライン展開では、短い関数の呼び出しを関数本体で置き換えます。関数呼び出しの準備や戻り処理が減るため速くなることがあります。一方で、展開しすぎるとコードサイズが大きくなるため、コンパイラは状況を見て判断します。

---

## 5. コード生成（Code Generation）

最適化後の中間表現を **アセンブリ言語** に変換します。

```c
int add(int a, int b) {
    return a + b;
}
```

↓ x86-64 アセンブリ（の一例）

```asm
add:
    push rbp
    mov  rbp, rsp
    mov  eax, edi      ; 引数 a を eax に
    add  eax, esi      ; eax += b
    pop  rbp
    ret                ; 結果 eax を返す
```

アセンブリはCPU命令に近い低水準の表現です。この例では、引数 `a` が `edi`、引数 `b` が `esi` に入り、`eax` に結果を置いて返しています。C言語の `return a + b;` という1行も、CPUから見るとレジスタへ値を移し、加算し、戻るという命令列になります。

---

## 6. アセンブルとリンク

**アセンブル：** アセンブリをバイナリのオブジェクトファイル（`.o`）に変換します。

**リンク：** 複数のオブジェクトファイルとライブラリを結合して実行ファイルを作ります。

```bash
# gcc は内部でこれらを全部やってくれる
gcc -c main.c    # → main.o（コンパイル+アセンブル）
gcc main.o -o main  # → main（リンク）
```

`gcc -c main.c` は実行ファイルまでは作らず、オブジェクトファイル `main.o` を作ります。`gcc main.o -o main` は、オブジェクトファイルと必要なライブラリを結合して、実行可能な `main` を作ります。複数ファイルのCプログラムでは、この分割が重要になります。

**動的リンクと静的リンク：**

| | 静的リンク | 動的リンク |
|--|---------|---------|
| 方法 | 実行ファイルにライブラリを埋め込む | 実行時にライブラリを読み込む |
| ファイルサイズ | 大きい | 小さい |
| 配布の容易さ | 容易（依存関係なし） | ライブラリが必要 |
| 例 | Go のデフォルト | C の共有ライブラリ (.so / .dll) |

## Python の実行モデル

Python（CPython）は以下の手順で実行されます。

hello.py
↓ 字句解析・構文解析
AST
↓ バイトコードコンパイル
hello.pyc（__pycache__/）
↓ CPython 仮想マシン（PVM）が解釈・実行
**バイトコード：**  
機械語ではなく、Python 仮想マシンが理解できる中間言語です。直接 CPU では実行できません。

```python
import dis

def add(a, b):
    return a + b

dis.dis(add)
```

`dis` は、Python関数がどのようなバイトコードに変換されているかを見るための標準ライブラリです。普段は意識しませんが、Pythonも実行前に一度バイトコードへ変換されています。

出力例：
```console
  2           0 LOAD_FAST      0 (a)
              2 LOAD_FAST      1 (b)
              4 BINARY_ADD
              6 RETURN_VALUE
```

`LOAD_FAST` はローカル変数を読み、`BINARY_ADD` は加算し、`RETURN_VALUE` は結果を返します。CPUの機械語ではなく、CPython仮想マシンが解釈する命令列です。このため、PythonはCのようなネイティブ実行ファイルとは違う実行モデルになります。

**`.pyc` ファイルとは？**  
`__pycache__/` 内に自動生成されるコンパイル済みバイトコードです。次回同じファイルを実行するとき、バイトコードのコンパイルをスキップして高速起動できます。

## JavaScript の V8 エンジン

Chrome と Node.js が使う JavaScript エンジンです。JIT コンパイルによって高速実行を実現します。

JS ソース
↓ パース → AST 生成
↓ Ignition（インタープリタ）がバイトコードとして実行
↓ ホットスポット（よく実行されるコード）を検出
↓ TurboFan（JIT コンパイラ）が機械語に変換
↓ 次回以降は機械語で高速実行
**型情報の最適化：**

```js
function add(a, b) { return a + b; }

add(1, 2);      // 整数の加算と判断して最適化
add(1.5, 2.5);  // float に変更 → 最適化をやり直し（Deoptimize）
add("a", "b");  // 文字列の結合に変更 → また Deoptimize
```

TypeScript を使うと型が固定されるため JIT の最適化が効きやすくなります。

JITは、実行中の値の傾向を見て最適化します。最初に数値だけが渡されていれば、数値加算として高速化できます。しかし途中で文字列が来ると、同じ `+` でも文字列結合の意味になり、前提が崩れます。このように最適化を取り消すことを Deoptimize と呼びます。

## 実際にやってみる：GCC のオプション

```bash
gcc hello.c -o hello       # 通常コンパイル
gcc hello.c -O2 -o hello   # 最適化レベル 2（本番用）
gcc hello.c -S             # アセンブリを出力（hello.s）
gcc hello.c -E             # プリプロセッサの出力のみ
gcc hello.c -g -o hello    # デバッグ情報を含める
```

`-O2` はよく使われる最適化レベルです。`-S` はアセンブリを出力するので、Cコードがどのような低水準命令に変換されるかを観察できます。`-E` はプリプロセッサ後の結果だけを出し、`#include` や `#define` が展開された姿を確認できます。`-g` はデバッガで行番号や変数名を見られるようにするオプションです。

アセンブリを見るとコンパイラが何をしているかが直接確認できます：

```bash
gcc -O0 -S hello.c && cat hello.s   # 最適化なし
gcc -O2 -S hello.c && cat hello.s   # 最適化あり（コードが短くなる）
```

`-O0` は最適化なしなので、元のCコードに近い形のアセンブリになりやすいです。`-O2` では不要な処理が消えたり、関数が展開されたりします。同じプログラムでも、コンパイルオプションによって生成されるコードが変わることを確認できます。
