データエンジニアリング¶

データサイエンティストが分析できる状態にデータを届けるための技術領域です。生のデータを収集・変換・蓄積・配信するパイプラインの設計と実装を扱います。MLOps や ML パイプラインとは区別される、データ基盤の構築に特化したスキルセットです。

はじめて読む人へ¶

データサイエンティストが「データを分析する」前に、誰かがデータを収集・整形・格納する仕事をしています。それがデータエンジニアリングです。適切なデータ基盤がなければ、高精度なモデルも作れません。

読む前に押さえること¶

  • ETL（Extract・Transform・Load）は、データを取り出して・加工して・格納する基本の流れです。
  • データウェアハウスは「分析専用の DB」で、日次・週次での大量データ集計に最適化されています。
  • dbt は SQL でデータ変換を管理するツールで、分析用テーブルの定義をコードとして管理できます。

読み終えたら説明できること¶

  • ETL と ELT の違いを説明できる。
  • データレイクとデータウェアハウスをどう使い分けるか説明できる。
  • dbt を使ってデータ変換を定義できる。

データエンジニアリングとデータサイエンスの違い¶

生データ（ログ・DB・API） ↓ [データエンジニアリング] ・収集・転送・変換・格納 ・パイプラインの設計・保守 ・データ品質・スキーマ管理 ↓ 整形済みデータ（DWH・データマート） ↓ [データサイエンス] ・分析・可視化・モデリング ・仮説検証・意思決定支援 現場ではデータサイエンティストがデータエンジニアリングも担当するケース（小規模チーム）と、専任チームが分かれているケース（大規模組織）があります。

ETL と ELT¶

ETL（Extract → Transform → Load）¶

変換してから格納する古典的な方式です。

import pandas as pd
import sqlalchemy

# Extract：ソース DB からデータを取得
engine_src = sqlalchemy.create_engine("postgresql://src_db")
df = pd.read_sql("SELECT * FROM orders WHERE created_at >= '2024-01-01'", engine_src)

# Transform：データを加工
df["total_price"] = df["quantity"] * df["unit_price"]
df["created_date"] = pd.to_datetime(df["created_at"]).dt.date
df = df.dropna(subset=["user_id"])
df = df[df["status"] == "completed"]

# Load：変換済みデータをデータウェアハウスへ格納
engine_dwh = sqlalchemy.create_engine("bigquery://project/dataset")
df.to_sql("sales_summary", engine_dwh, if_exists="append", index=False)

ELT（Extract → Load → Transform）¶

まず生データを格納し、DWH の中で変換します。BigQuery・Snowflake 時代の主流です。

生データ → DWH（ステージング領域に格納）→ SQL/dbt で変換 → 分析用テーブル ELT のメリット： - DWH の高い計算能力を変換に使える - 生データが残るので再変換が可能（やり直しがきく） - 変換ロジックを SQL / dbt で管理できる

データウェアハウスとデータレイク¶

データウェアハウス（DWH）¶

分析用に最適化された DB です。大量の集計クエリを高速に処理するために、列指向ストレージを使います。

サービス 特徴
BigQuery（GCP） サーバーレス。クエリ量課金。ML 機能内蔵
Snowflake マルチクラウド。コンピュートとストレージを分離
Redshift（AWS） Postgres 互換。EC2 ベース
DuckDB ローカル・組み込み。分析に特化した軽量 OLAP 
-- BigQuery の例：直近 30 日の売上集計
SELECT
    DATE_TRUNC(created_at, MONTH) AS month,
    product_category,
    SUM(amount) AS total_sales,
    COUNT(DISTINCT user_id) AS unique_buyers
FROM `project.dataset.orders`
WHERE created_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
  AND status = 'completed'
GROUP BY 1, 2
ORDER BY 1 DESC, total_sales DESC;

データレイク¶

生データを形式を変えずに大量に蓄積するストレージです（S3、GCS、ADLS）。

データレイク（生データ）:

データレイク（生データ）
  ├── /raw/clickstream/2024/01/01/*.json    ← アプリログ
  ├── /raw/db_snapshot/users/20240101.csv  ← DB のスナップショット
  └── /raw/api_response/weather/*.json     ← 外部 API レスポンス

変換を経て、整形済みデータがデータウェアハウスに格納されます。

データウェアハウス（整形済み）:

データウェアハウス（整形済み）
  ├── staging.clickstream        ← パース済み
  ├── staging.users              ← 型変換済み
  └── marts.sales_daily          ← 集計済み・分析用

メダリオンアーキテクチャ¶

データレイクを 3 層に分けて管理する現代的な設計パターンです。

Bronze（生データ）→ Silver（クレンジング済み）→ Gold（分析用マート）

Bronze： - ソースから取り込んだ生データそのまま - スキーマは変えない。削除もしない

Silver： - 型変換・欠損処理・重複排除 - 正規化されたテーブル

Gold： - ビジネスロジックを適用した集計テーブル - BIツール・DS・機械学習が直接使う

dbt（data build tool）¶

データウェアハウス内の変換ロジックを SQL でバージョン管理 するツールです。

プロジェクト構成：
  dbt_project/
  ├── models/
  │   ├── staging/
  │   │   ├── stg_orders.sql       ← 生データのクレンジング
  │   │   └── stg_users.sql
  │   └── marts/
  │       └── sales_daily.sql      ← 集計・変換
  ├── tests/
  │   └── schema.yml               ← データ品質テスト定義
  └── dbt_project.yml

-- models/staging/stg_orders.sql
SELECT
    order_id,
    user_id,
    CAST(amount AS FLOAT64) AS amount,
    TIMESTAMP(created_at) AS created_at,
    LOWER(TRIM(status)) AS status
FROM {{ source('raw', 'orders') }}
WHERE order_id IS NOT NULL


-- models/marts/sales_daily.sql
SELECT
    DATE(created_at) AS date,
    COUNT(order_id) AS order_count,
    SUM(amount) AS total_sales
FROM {{ ref('stg_orders') }}   -- stg_orders.sql を参照
WHERE status = 'completed'
GROUP BY 1

# tests/schema.yml（データ品質テスト）
models:
  - name: stg_orders
    columns:
      - name: order_id
        tests:
          - not_null
          - unique
      - name: status
        tests:
          - accepted_values:
              values: ['completed', 'pending', 'cancelled']

dbt run         # 変換を実行
dbt test        # データ品質テストを実行
dbt docs serve  # データカタログをブラウザで表示

Apache Spark（大規模データ処理）¶

単一マシンに収まらない TB〜PB 規模のデータを並列処理するための分散計算フレームワークです。

from pyspark.sql import SparkSession
from pyspark.sql import functions as F

spark = SparkSession.builder.appName("SalesAggregation").getOrCreate()

# 大規模な Parquet ファイルを読み込む（実際には分散ストレージ上）
df = spark.read.parquet("gs://bucket/raw/orders/")

# 変換（遅延評価：run() まで実行されない）
result = (
    df
    .filter(F.col("status") == "completed")
    .withColumn("date", F.to_date("created_at"))
    .groupBy("date", "product_category")
    .agg(
        F.sum("amount").alias("total_sales"),
        F.count("order_id").alias("order_count")
    )
    .orderBy("date")
)

# 結果を BigQuery に書き込む
result.write.format("bigquery").option("table", "dataset.sales_daily").save()

ストリーミング処理¶

バッチ処理（まとめて処理）に対し、データが発生したリアルタイムで処理します。

Apache Kafka¶

分散メッセージングシステムです。データの「流れ」をトピックで管理します。

[Producer（ログサーバー）] → [Kafka トピック] → [Consumer（集計サービス）] └→ [Consumer（アラートサービス）] 

from kafka import KafkaProducer, KafkaConsumer
import json

# Producer：イベントを送信
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)
producer.send('user-events', {'user_id': 123, 'action': 'purchase', 'amount': 4980})

# Consumer：イベントを受信して処理
consumer = KafkaConsumer(
    'user-events',
    bootstrap_servers=['localhost:9092'],
    value_deserializer=lambda m: json.loads(m.decode('utf-8'))
)
for message in consumer:
    event = message.value
    print(f"User {event['user_id']} bought {event['amount']} yen")

データ品質とデータリネージ¶

データ品質の観点¶

観点 意味 チェック例
完全性 欠損がないか NULL 率が閾値以下か
一意性 重複がないか PK の重複チェック
適時性 期待した時間に更新されているか 最終更新日のモニタリング
妥当性 値が許容範囲内か 年齢が 0〜150 の範囲か
一貫性 テーブル間で矛盾がないか orders の user_id が users に存在するか

Great Expectations（データ品質ツール）¶

import great_expectations as gx

context = gx.get_context()
validator = context.sources.pandas_default.read_csv("orders.csv")

# データ品質の期待値を定義
validator.expect_column_values_to_not_be_null("order_id")
validator.expect_column_values_to_be_unique("order_id")
validator.expect_column_values_to_be_between("amount", min_value=0, max_value=1_000_000)
validator.expect_column_values_to_be_in_set("status", ["completed", "pending", "cancelled"])

results = validator.validate()
print(results["success"])  # True / False

確認問題¶

  1. ETL と ELT の違いを説明し、ELT が主流になった理由を述べてください。
  2. データレイクとデータウェアハウスの使い分けを説明してください。
  3. dbt の {{ ref() }} 関数は何をしているか説明してください。
  4. バッチ処理とストリーミング処理を比較し、それぞれ適したユースケースを挙げてください。
