Web アプリの攻撃と防御¶

セキュリティ基礎 で「何が危ないか」の概要を掴んだら、このページでは 「攻撃がコードレベルでどう起きて、どう防ぐか」 を実装の視点から理解します。

はじめて読む人へ¶

セキュリティは「知っているかどうか」で大きく差がつく分野です。「プレースホルダって何？」「bcrypt って何で必要なの？」という疑問に、コードで答えます。

読む前に押さえること¶

  • セキュリティ基礎 — XSS・CSRF・.env の概要
  • データベース基礎 — SQL の基本的な書き方

読み終えたら説明できること¶

  • SQL インジェクションを「文字列結合 vs プレースホルダ」で説明できる
  • bcrypt が MD5 より安全な理由を説明できる
  • 認証（ログイン確認）と認可（権限確認）の違いをコードで説明できる

SQL インジェクション¶

「注文書を偽造される」攻撃¶

レストランのオーダーシートで考えてみましょう。

普通の注文 改ざんされた注文
「ハンバーガーを1つ」 「ハンバーガーを1つ、それと全テーブルの注文をキャンセルして」
SQL インジェクションはこれと同じことが起きます。 
# ❌ 危険：ユーザー入力をそのまま SQL に文字列結合
email = request.form["email"]
query = f"SELECT * FROM users WHERE email = '{email}'"

# ユーザーが email に以下を入力した場合:
# ' OR '1'='1
# → SELECT * FROM users WHERE email = '' OR '1'='1'
#   全ユーザーが取得できる（ログインバイパス）

# ' ; DROP TABLE users; --
# → SELECT * FROM users WHERE email = ''; DROP TABLE users; --'
#   users テーブルが消える

✅ 防御：プレースホルダ（パラメータバインディング）¶

# 安全：SQL の「構造」と「データ」を分離する
cursor.execute(
    "SELECT * FROM users WHERE email = %s",
    (email,)   # ← 値はここに入れる（SQL として解釈されない）
)

# SQLAlchemy の場合（ORM を使うとさらに安全）
from sqlalchemy import select
result = session.execute(
    select(User).where(User.email == email)
)

プレースホルダの仕組み：

プレースホルダを使うと、SQL の構造（骨格）とデータが完全に分離されます。? に何を入れても「データ」として解釈されるため、' OR '1'='1 のような入力も「email がその文字列の人を探す」という意味になるだけで、SQL の構造は変わりません。 覚えるルール： ユーザー入力を文字列結合で SQL に混ぜない。必ずプレースホルダ。

XSS（クロスサイトスクリプティング）¶

「掲示板に罠を仕掛ける」攻撃¶

コメント欄のある掲示板を想像してください。

1. 攻撃者がコメントに以下を投稿

  1. サーバーがこれをそのままページに埋め込む

  2. 他のユーザーがページを開く → ブラウザが「コメントのテキスト」ではなく「JavaScript」として実行

  3. そのユーザーのクッキー（ログインセッション）が攻撃者に送信される

  4. 攻撃者がそのクッキーでそのユーザーとしてログインできる

✅ 防御：エスケープ¶

エスケープとは、HTML の特殊文字を無害な文字列に変換することです。

from markupsafe import escape

# ❌ 危険：そのまま出力
return f"<p>コメント: {user_comment}</p>"

# ✅ 安全：エスケープしてから出力
return f"<p>コメント: {escape(user_comment)}</p>"

# <script>alert(1)</script> が入力されたとき:
# エスケープ前: <script>alert(1)</script>  ← 実行される
# エスケープ後: &lt;script&gt;alert(1)&lt;/script&gt;  ← テキストとして表示

// React は {} で囲んだ変数を自動エスケープ（安全）
return <p>{userComment}</p>

// ❌ これだけは危険（エスケープを意図的に無効化）
return <p dangerouslySetInnerHTML={{__html: userComment}} />

追加対策：CSP ヘッダー

# FastAPI でのCSP設定
from fastapi import Response

@app.middleware("http")
async def add_security_headers(request, call_next):
    response = await call_next(request)
    response.headers["Content-Security-Policy"] = (
        "default-src 'self'; "
        "script-src 'self'; "  # 自分のオリジンのスクリプトだけ許可
        "object-src 'none'"
    )
    return response

CSP は「このページで動かしてよいスクリプトの出所」をブラウザに伝えます。万が一 XSS が刺さっても、外部への送信をブロックできます。

CSRF（クロスサイトリクエストフォージェリ）¶

「ログインしたままの状態を悪用される」攻撃¶

あなたが銀行にログイン中（ブラウザにクッキーあり）、攻撃者が以下のページを用意してあなたに踏ませます。

<html>
  <body onload="document.getElementById('f').submit()">
    <form id="f" action="https://your-bank.com/transfer" method="POST">
      <input name="to"     value="攻撃者の口座">
      <input name="amount" value="100000">
    </form>
  </body>
</html>

あなたがそのページを開いた瞬間、フォームが自動送信されます。ブラウザは「銀行サイトへのリクエスト」に自動でクッキーを付けるため、銀行から見ると「あなた本人が振込した」ように見えます。 XSS との違い： - XSS → 攻撃者のコードがあなたのブラウザで動く - CSRF → あなたのブラウザが攻撃者の意図したリクエストを送る

✅ 防御1：CSRF トークン¶

# サーバーがランダムなトークンをフォームに埋め込む
import secrets

@app.get("/transfer-form")
def show_form(request: Request):
    token = secrets.token_hex(32)
    request.session["csrf_token"] = token
    return templates.TemplateResponse("transfer.html", {"token": token})

@app.post("/transfer")
def do_transfer(request: Request, data: TransferRequest):
    # フォームから送られてきたトークンとセッションのトークンを比較
    if data.csrf_token != request.session.get("csrf_token"):
        raise HTTPException(status_code=403, detail="CSRF トークンが不正です")
    # ... 処理続行

なぜ有効か： 攻撃者の罠ページはあなたの銀行サイトの CSRF トークンを読めない（同一オリジンポリシー）ので、正しいトークン付きのリクエストを作れません。

✅ 防御2：SameSite Cookie（最も簡単）¶

# クッキーを「同じサイトからのリクエストにしか送らない」設定
response.set_cookie(
    "session",
    value=session_id,
    samesite="Strict",   # 別のサイトからのリクエストにはクッキーを付けない
    httponly=True,        # JavaScript からクッキーを読めなくする（XSS 対策にも）
    secure=True           # HTTPS のみで送信
)

パスワードの安全な保存¶

❌ やってはいけないこと¶

# 1. 平文保存（論外）
user.password = "abc123"

# 2. MD5・SHA-1（高速すぎる → ブルートフォースで簡単に破られる）
import hashlib
user.password = hashlib.md5(password.encode()).hexdigest()
# MD5 は 1秒間に数十億回の計算ができる → 辞書攻撃で即破られる

✅ bcrypt / Argon2 を使う¶

from passlib.context import CryptContext

# パスワードハッシュの設定（bcrypt を推奨）
pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# 保存時: 平文 → ハッシュ（DB に保存するのはこれだけ）
hashed = pwd_context.hash("user_password")
# → "$2b$12$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"

# 検証時: 入力パスワードとハッシュを比較
is_valid = pwd_context.verify("user_password", hashed)

bcrypt が安全な理由：

MD5 は非常に高速（GPU で 1 秒間に数十億回計算可能） で、ソルトもないため同じパスワードは必ず同じハッシュになります。レインボーテーブル（ハッシュ逆引き表）で即座に解読されます。

bcrypt は意図的に遅くできる（コスト係数で調整）ため、GPU を使っても 1 パスワードの解読に秒単位かかります。また自動でランダムなソルトが付与されるため、同じパスワードでも毎回異なるハッシュになり、大量解読が実用的でなくなります。

認証 vs 認可¶

混同しやすいので整理します。

認証（Authentication）= 「あなたは誰？」を確認 → ログイン（ID・パスワードの確認）

認可（Authorization）= 「あなたはこれをしていい？」を確認 → アクセス制御（このユーザーはこのページを見ていいか） 

# 認証: ログインしているかどうかだけ確認
@app.get("/dashboard")
def dashboard(current_user = Depends(get_current_user)):
    return {"user": current_user}

# 認可: ログインしていて、さらに「自分のリソース」かどうか確認
@app.get("/users/{user_id}/data")
def get_user_data(user_id: int, current_user = Depends(get_current_user)):
    # 認可チェック: 自分のデータ以外は見せない
    if current_user["id"] != user_id:
        raise HTTPException(status_code=403, detail="アクセス権限がありません")
    return db.get_user_data(user_id)

よくある間違い： 認証（ログイン確認）だけして、認可（権限確認）を忘れる。URL の user_id を変えると他人のデータが見えてしまう（水平権限昇格）。

RBAC（ロールベースアクセス制御）¶

from enum import Enum

class Role(str, Enum):
    ADMIN  = "admin"
    EDITOR = "editor"
    VIEWER = "viewer"

def require_role(required_role: Role):
    def checker(current_user = Depends(get_current_user)):
        if current_user["role"] != required_role:
            raise HTTPException(status_code=403, detail="権限がありません")
        return current_user
    return checker

# 管理者だけがアクセスできるエンドポイント
@app.delete("/users/{user_id}")
def delete_user(user_id: int, _=Depends(require_role(Role.ADMIN))):
    db.delete_user(user_id)

攻撃と防御の早見表¶

攻撃 何をされるか 防御策
SQL インジェクション DB の全データ盗取・削除・改ざん プレースホルダを使う
XSS ユーザーのクッキー盗取・画面改ざん HTML エスケープ・CSP ヘッダー
CSRF ユーザーが意図しない操作（振込など） CSRF トークン・SameSite Cookie
パスワード漏洩 平文 DB 流出でパスワード特定 bcrypt/Argon2 でハッシュ保存
水平権限昇格 他人のデータが見える リクエストごとの認可チェック
セッションハイジャック クッキーを盗まれてなりすまし HttpOnly・Secure Cookie・短い有効期限

確認問題¶

  1. query = f"SELECT * FROM users WHERE id = {user_id}" の何が問題か、攻撃者がどんな入力をしたら何が起きるかを具体的に説明してください。
  2. なぜ MD5 ではなく bcrypt でパスワードをハッシュする必要があるのか、「速さ」の観点から説明してください。
  3. 認証（Authentication）と認可（Authorization）の違いを、「ログインはできたけど他人のデータが見えた」というシナリオを使って説明してください。
