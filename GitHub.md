# GitHub

## GitHub とは

**Git で管理しているファイルをインターネット上に保存・共有できるサービス** です。

Git がローカル（自分の PC）でのバージョン管理ツールだとすると、GitHub はその保存場所をクラウドに用意してくれるサービスです。

> **Google ドライブで例えると：**  
> 自分の PC 上のファイルを Google ドライブに保存すると、他の人と共有できますよね。  
> GitHub はコードやテキストファイルに特化した「Google ドライブ」のようなものです。

主な機能：
- コードをオンラインで保存・共有します
- 変更内容をチームでレビューします（**Pull Request**）
- 変更をきっかけに自動でサイトを更新します

---

## はじめて読む人へ

GitHub は、Git の履歴をチームで共有する場所です。自分のPCだけでなく、メンバー全員が同じ変更履歴を見られるようになります。

コードやコマンドが出てきたら、最初から全部を覚えようとしなくて大丈夫です。まずは「何を入力し、何が処理され、何が出力されるのか」を文章で説明できるように読むと、手を動かす前の理解が安定します。

### 読む前に押さえること

- Pull Request は、変更を本流に入れる前の相談場所です。
- レビューは人を責める場ではなく、品質を一緒に上げる場です。
- コンフリクトは、同じ場所を複数人が編集したときに起きます。

### 読み終えたら説明できること

- リポジトリ、Issue、Pull Request の役割を説明できる。
- PR を作ってレビューを受ける流れを理解できる。
- コンフリクト時に何を確認すべきか分かる。

## リポジトリのクローン（初回だけ）

**クローン** とは、GitHub 上のリポジトリを自分の PC にまるごとコピーすることです。

初回参加時は、まずリモートリポジトリを手元に取得します。`git clone` は、ファイルだけでなく Git の履歴も一緒にダウンロードします。

```bash
git clone https://github.com/ohmitechacademy/ota_hp.git
```

このコマンドを実行すると、`ota_hp` というフォルダが作られ、中にすべてのファイルが入ります。

クローン後は、そのフォルダの中で作業します。以後の `git status`、`git switch`、`git commit` などは、基本的にこのリポジトリ内で実行します。

## Pull Request とは

Pull Request は、変更を main ブランチに入れる前に、他の人に見てもらうための仕組みです。単なる承認ボタンではなく、「この変更で何をしたか」「なぜ必要か」「どこを確認してほしいか」を共有する場所です。

初学者は、PR を作ることを大げさに考えなくて大丈夫です。小さな変更でも PR にすると、差分を見る練習になり、コードレビューの会話を通じて設計や書き方を学べます。

**Pull Request（PR）** は、「自分が加えた変更を、チームに確認してもらってからメインのブランチに取り込む」ための仕組みです。

直接 `main` ブランチを変更するのではなく、一度 PR を通すことで：
- 誰かに確認してもらえます（レビュー）
- ミスを防げます
- 変更の記録が残ります

### Pull Request の流れ

次の図は、作業ブランチで変更した内容が、本番サイトへ反映されるまでの流れです。PR は、変更を `main` へ入れる前に確認するための中間地点です。

```
① feature ブランチで作業・コミット・プッシュします
      ↓
② GitHub で Pull Request を作成します
      ↓
③ チームメンバーがコードを確認します（レビュー）
      ↓
④ 問題なければ「Merge」して main に取り込みます
      ↓
⑤ Cloudflare Pages が変更を検知して自動ビルド・デプロイします
      ↓
⑥ 本番サイト（ohmitechacademy.com）に反映が完了します
```

PR の本文には、何を変更したか、なぜ変更したか、どこを確認してほしいかを書きます。レビュアーは差分を見て、動作や設計に問題がないか確認します。

> **「デプロイ」とは：** コードをインターネット上のサーバーに置き、誰でもアクセスできる状態にすることです。このプロジェクトでは **main ブランチへのマージが自動的にデプロイをトリガーします**。手動でサーバーにファイルをアップロードする作業は不要です。詳しくは [Cloudflare](Cloudflare) ページで解説しています。

### Pull Request の作り方

1. `git push` でブランチを GitHub にアップロードします
2. GitHub のリポジトリページを開くと、黄色いバナーで **「Compare & pull request」** ボタンが表示されます
3. クリックして変更内容の説明を書きます
4. **「Create pull request」** をクリックします

レビュアーがコメントや承認をしたら **「Merge pull request」** → **「Confirm merge」** でマージ完了です。

## Pull Request のレビュープロセス詳細

### レビュアーができること

PR を受け取ったレビュアーは **「Files changed」** タブでコードの差分を確認し、3 つのアクションを取れます。

| アクション | 意味 |
|-----------|------|
| **Comment** | コメントのみ（承認でも却下でもない） |
| **Approve** | 変更を承認（マージ OK） |
| **Request changes** | 修正を要求（マージ不可） |

### 「Request changes」を受けたとき

```
① レビュアーが修正コメントを付けて「Request changes」します
      ↓
② 自分のブランチでコードを修正してコミット・プッシュします
   （同じブランチに push すれば自動で PR に反映されます）
      ↓
③ コメントに返信して「Resolve conversation」をクリックします
      ↓
④ レビュアーに再レビューを依頼します
      ↓
⑤ 「Approve」をもらえたらマージします
```

Request changes は怒られているという意味ではなく、「main に入れる前にここを直そう」という合図です。同じブランチへ追加コミットを push すれば、PR は自動で更新されます。

### レビューコメントへの返信マナー

- 修正した場合：「修正しました。ご確認ください。」
- 同意できない場合：理由を添えて丁寧に説明します
- 疑問がある場合：「この意図は〇〇ですが、どのように修正すべきでしょうか？」

## マージコンフリクトの解決

### コンフリクトとは

同じファイルの同じ行を、複数の人が別々に編集したときに発生する「衝突」です。Git が自動でマージできず、手動での解決が必要になります。

次の記号は、Git が「どちらの変更を採用すべきか分からない」と示している部分です。`HEAD` 側と `main` 側の変更を見比べ、最終的に残したい内容へ手で直します。

```
<<<<<<< HEAD（自分のブランチの変更）
<h1>近江テック・アカデミー</h1>
=======
<h1>近江テックアカデミー（公式）</h1>
>>>>>>> main（マージ先の変更）
```

### コンフリクトの解決手順

コンフリクト解決では、まず `main` の最新を取り込み、衝突したファイルを編集し、解決済みとしてコミットします。

```bash
# 1. main ブランチの最新を取り込みます
git pull origin main

# → コンフリクトが発生したファイルが表示されます
# CONFLICT (content): Merge conflict in src/components/Header.astro
```

この時点では、Git がファイル内に衝突マーカーを書き込んでいます。次にエディタでそのファイルを開きます。

```bash
# 2. コンフリクトしたファイルを VS Code などで開きます
# <<<<<<< 〜 >>>>>>> の記号を含む箇所を探して手動で修正します

# 正しい内容に書き直します（記号ごと削除します）
# 例：<h1>近江テック・アカデミー</h1>
```

衝突マーカーは最終ファイルに残してはいけません。両方の変更を統合するのか、片方だけを残すのかを判断して、通常のコードや文章に戻します。

```bash
# 3. 解決済みとしてステージングします
git add src/components/Header.astro

# 4. コミットします（コミットメッセージは自動で入力される場合もあります）
git commit -m "コンフリクトを解決"

# 5. プッシュします
git push origin feature/xxx
```

`git add` は「このファイルのコンフリクトは解決しました」と Git に伝える役割もあります。最後に push すると、PR 上のコンフリクト状態が更新されます。

### コンフリクトを防ぐには

- 作業前に必ず `git pull origin main` で最新を取り込みます
- ブランチを長期間放置しません（こまめにマージします）
- 同じファイルを複数人が同時に編集しないようチームで調整します

## GitHub Pages

リポジトリのファイルをそのまま静的サイトとして無料で公開できる機能です。

> このプロジェクトでは **Cloudflare Pages** を使っているため、GitHub Pages は使っていません。  
> 仕組みは似ているので参考として知っておいてください。

### 設定方法

1. リポジトリの **Settings** → **Pages** を開きます
2. **Branch** で `main` を選択して **Save** します
3. 数分後に `https://ohmitechacademy.github.io/ota_hp/` で公開されます

## GitHub Wiki

リポジトリに紐づいたドキュメントサイトを作れる機能です。今あなたが読んでいるこのページがまさに GitHub Wiki です。

Markdown（`#` で見出し、`**` で太字など）という書式でページを書きます。

### Web ブラウザから編集する方法

1. Wiki のページを開きます
2. 右上の **「Edit」** をクリックします
3. テキストを編集します
4. **「Save Page」** をクリックします

### ターミナルから編集する方法

Wiki も Git リポジトリとしてクローンできます。ローカルのエディタで複数ページをまとめて編集したいときは、この方法が便利です。

```bash
# Wiki リポジトリをクローンします（初回だけ）
git clone https://github.com/ohmitechacademy/tech-guide.wiki.git
cd tech-guide.wiki

# ファイルを編集後
git add .
git commit -m "Git ページの内容を追記"
git push origin master
```

通常のコードリポジトリと同じように、Markdown ファイルを編集し、コミットして push します。違いは、Wiki 専用リポジトリではブランチ名が `master` である点です。

> **`master` ブランチについて**  
> ふだんのコード管理では `main` ブランチを使いますが、GitHub Wiki は GitHub が管理する別リポジトリで、デフォルトが `master` になっています。  
> これは GitHub の仕様であり、間違いではありません。Wiki 専用の話として覚えておいてください。

## よくある疑問

**Q. GitHub と Git は別物？**  
A. はい。Git はツール（バージョン管理の仕組み）、GitHub はサービス（Git リポジトリのホスティングサービス）です。Git は GitHub がなくても使えます。

**Q. アカウントが必要？**  
A. 読むだけなら不要です。コードのプッシュや PR の作成にはアカウントが必要です。
