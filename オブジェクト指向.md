> OOP の 4 原則を理解しておくと、次の [ソフトウェア設計](ソフトウェア設計) で学ぶ責務分離・DI・テスト設計の意図が明確になります。

## はじめて読む人へ

オブジェクト指向は、データと処理をひとまとまりの部品として扱う考え方です。現実のものをそのまま写すというより、プログラムの責務を分けて整理するために使います。

コードやコマンドが出てきたら、最初から全部を覚えようとしなくて大丈夫です。まずは「何を入力し、何が処理され、何が出力されるのか」を文章で説明できるように読むと、手を動かす前の理解が安定します。

### 読む前に押さえること

- クラスは設計図、インスタンスは実体です。
- カプセル化は、内部状態を安全に扱うための考え方です。
- 継承よりも、必要な部品を組み合わせるコンポジションが適切な場合も多いです。

### 読み終えたら説明できること

- クラス、インスタンス、継承、ポリモーフィズムを説明できる。
- is-a と has-a の違いを理解できる。
- SOLID原則の概要を説明できる。

## OOP の 4 原則

| 原則 | 意味 |
|------|------|
| カプセル化 | データと操作をまとめ、内部を隠します |
| 継承 | 既存クラスの機能を引き継いで拡張します |
| ポリモーフィズム | 同じインターフェースで異なる動作をさせます |
| 抽象化 | 共通の特徴を抽出し、詳細を隠します |

オブジェクト指向は「現実世界をそのままクラスにする技術」ではありません。大事なのは、変更されやすい部分と安定している部分を分け、プログラムの責務を整理することです。たとえば、通知を送る処理では「注文が完了したら通知する」というルールは安定していますが、「メールで送るのか、Slackで送るのか」は変わる可能性があります。このような違いをクラスやインターフェースで分けると、後から変更しやすくなります。

## クラスとインスタンス

クラスは、データと処理をまとめた設計図です。インスタンスは、その設計図から作られた実体です。たとえば `User` クラスが設計図で、`田中さんのユーザー情報` がインスタンスにあたります。

クラスを使うと、関連するデータと操作を近くに置けます。ユーザー名、メールアドレス、権限確認の処理がバラバラに散らばるより、1つの概念としてまとまっている方が読みやすくなります。

次のコードでは、`__init__` がインスタンスを作るときの初期化処理です。`self` は「いま操作しているインスタンス自身」を表します。`self.name` や `self._email` に値を入れることで、そのインスタンスが持つデータとして保存されます。

```python
class User:
    def __init__(self, name: str, email: str):
        self.name  = name
        self._email = email   # _ は「外から触らないでほしい」慣習

    def greet(self) -> str:
        return f"こんにちは、{self.name} さん"

    @property
    def email(self) -> str:
        return self._email

    @email.setter
    def email(self, value: str):
        if "@" not in value:
            raise ValueError("メールアドレスの形式が正しくありません")
        self._email = value


user = User("山田太郎", "yamada@example.com")
print(user.greet())   # こんにちは、山田太郎 さん
user.email = "new@example.com"  # setter で検証が走ります
```

`@property` を使うと、外からは `user.email` のように属性として読めますが、内部ではメソッドを通して値を返せます。`@email.setter` は値を代入するときに実行される処理です。この例では、メールアドレスに `@` が含まれているかを確認してから `_email` を更新しています。つまり、クラスの外側に「メールアドレスの検証ルール」を散らばらせず、`User` クラスの中に閉じ込めています。

## カプセル化

内部状態を直接変更させず、メソッドを通じて操作させることで、不正な状態を防ぎます。

カプセル化の目的は、単に変数を隠すことではありません。「そのオブジェクトが壊れた状態にならないように、変更の入口を限定する」ことです。銀行口座なら、残高が勝手にマイナスになったり、0円以下の入金ができたりすると困ります。そこで、残高を直接書き換えさせず、`deposit` や `withdraw` のようなメソッド経由で操作させます。

```python
# 悪い例：外から直接変更できます
user.balance = -10000  # 残高がマイナスになっても気づけません

# 良い例：メソッド経由で検証します
class BankAccount:
    def __init__(self, balance: int = 0):
        self._balance = balance

    def deposit(self, amount: int):
        if amount <= 0:
            raise ValueError("入金額は正の数でなければなりません")
        self._balance += amount

    def withdraw(self, amount: int):
        if amount > self._balance:
            raise ValueError("残高が不足しています")
        self._balance -= amount

    @property
    def balance(self) -> int:
        return self._balance
```

この `BankAccount` では、外部から `_balance` を直接変更するのではなく、入金と出金のメソッドを使う想定です。`deposit` は正の金額だけ受け付け、`withdraw` は残高不足を防ぎます。クラスの中にルールを集めておくと、どこで口座残高が変わるのかを追いやすくなります。

## 継承（Inheritance）

既存クラス（親クラス）の属性・メソッドを引き継いで、新しいクラス（子クラス）を作ります。

継承は、共通する性質を親クラスに置き、具体的な違いを子クラスで表す仕組みです。次の例では、`Animal` は「名前を持ち、鳴くことができる」という共通の型を表し、`Dog` と `Cat` が具体的な鳴き方を実装しています。

```python
class Animal:
    def __init__(self, name: str):
        self.name = name

    def speak(self) -> str:
        raise NotImplementedError

class Dog(Animal):
    def speak(self) -> str:
        return f"{self.name}: ワン！"

class Cat(Animal):
    def speak(self) -> str:
        return f"{self.name}: ニャー"

dog = Dog("ポチ")
cat = Cat("タマ")
print(dog.speak())  # ポチ: ワン！
print(cat.speak())  # タマ: ニャー
```

`Animal.speak()` は `NotImplementedError` を投げています。これは「動物であれば鳴くはずだが、具体的な鳴き方は子クラスで決めてください」という意味です。親クラスに共通の入口を用意し、子クラスごとに詳細を変えることで、呼び出し側は `speak()` という同じ名前で扱えます。

**継承の注意点：**
- 「is-a 関係」（犬は動物 → Dog is-a Animal）が成立するときだけ使います
- 「has-a 関係」（車はエンジンを持つ → Car has-a Engine）は継承でなく**コンポジション**を使います
- 継承の深さが 3 以上になると管理が難しくなります（継承より合成を好みます）

## ポリモーフィズム（Polymorphism）

同じメソッド名で、クラスに応じた異なる動作をさせます。

ポリモーフィズムは、「呼び出し側が具体的なクラス名を知らなくても、同じ操作を呼べる」ことです。`Dog` か `Cat` かを `if` 文で判定して処理を分けるのではなく、どちらも `speak()` を持っているという前提で扱います。

```python
animals: list[Animal] = [Dog("ポチ"), Cat("タマ"), Dog("コロ")]

for animal in animals:
    print(animal.speak())   # 型を気にせず同じコードで呼べます
# ポチ: ワン！
# タマ: ニャー
# コロ: ワン！
```

ポリモーフィズムにより、新しい動物クラスを追加しても `for` ループのコードを変更する必要がありません（**開放閉鎖原則 OCP**）。

たとえば後から `Bird` クラスを追加しても、`Bird` が `speak()` を実装していれば、ループ側はそのまま動きます。変更に強い設計では、「新しい種類を追加するたびに既存の分岐を編集する」状態を減らします。

## 抽象クラスとインターフェース

**抽象クラス：** インスタンス化できず、必ずサブクラスで実装すべきメソッドを定義します。

抽象クラスは、具体的な実装ではなく「この形のメソッドを必ず持ってください」という約束を表します。次の `Notifier` は、通知の送り方そのものは決めません。ただし、どの通知クラスも `send(message)` を持つことを要求します。

```python
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message: str) -> None:
        """通知を送る（実装は子クラスに任せます）"""

class EmailNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"メール送信: {message}")

class SlackNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"Slack 送信: {message}")
```

`EmailNotifier` と `SlackNotifier` は送信方法が違いますが、どちらも `send` という同じメソッドを持っています。呼び出し側は「メールかSlackか」を細かく知らなくても、`notifier.send(...)` と書けます。

```python
# 依存性の注入と組み合わせます
class OrderService:
    def __init__(self, notifier: Notifier):
        self.notifier = notifier

    def place_order(self, order):
        # 注文処理 ...
        self.notifier.send(f"注文 {order.id} を受け付けました")

# 本番はメール、テストは Mock に差し替えられます
service = OrderService(EmailNotifier())
```

`OrderService` は、具体的な `EmailNotifier` に直接依存するのではなく、`Notifier` という抽象に依存しています。この形にすると、本番ではメール通知、テストでは偽物の通知、別サービスではSlack通知のように差し替えられます。これが後のソフトウェア設計で学ぶ DI（依存性注入）につながります。

## is-a vs has-a

継承を使うべきか、部品として持つべきかを考えるときは、`is-a` と `has-a` の違いを見ると判断しやすくなります。`ElectricCar` は `Car` の一種なので `is-a` 関係です。一方、`Car` は `Engine` の一種ではなく、`Engine` を持っているので `has-a` 関係です。

```python
# is-a → 継承（ElectricCar IS-A Car）
class Car:
    def drive(self): ...

class ElectricCar(Car):
    def charge(self): ...

# has-a → コンポジション（Car HAS-A Engine）
class Engine:
    def start(self): ...

class Car:
    def __init__(self):
        self.engine = Engine()   # 持っています

    def drive(self):
        self.engine.start()
        ...
```

**「継承よりコンポジションを好む（Favor Composition over Inheritance）」** は OOP 設計の重要原則です。

コンポジションでは、必要な機能を「親から受け継ぐ」のではなく、「部品として持つ」ことで組み合わせます。部品を差し替えやすくなるため、テストや機能追加に強くなります。継承は便利ですが、親クラスの変更が子クラス全体に影響するため、関係が本当に `is-a` と言える場面に絞るのが安全です。

## SOLID 原則の概要

| 文字 | 名前 | 一言まとめ |
|------|------|-----------|
| S | 単一責任原則 | 1 クラス 1 つの責務 |
| O | 開放閉鎖原則 | 拡張に開き、修正に閉じる |
| L | リスコフ置換原則 | 子クラスは親クラスの代わりに使えます |
| I | インターフェース分離原則 | 使わないメソッドを強制しません |
| D | 依存性逆転原則 | 抽象に依存し、実装に依存しません |

## 面接でよく聞かれること

**Q. カプセル化の目的は何ですか？**  
→ 内部状態への不正アクセスを防ぎ、変更の影響範囲を限定するためです。

**Q. 継承とコンポジションの使い分けは？**  
→ is-a 関係が明確なときだけ継承します。「機能を追加したい」だけならコンポジションを使います。

**Q. ポリモーフィズムのメリットは？**  
→ 新しいクラスを追加しても呼び出し側のコードを変更しなくて済みます（OCP）。

**Q. 抽象クラスとインターフェースの違いは？**  
→ 抽象クラスは実装を持てる・状態を持てます。インターフェース（Python では Protocol/ABC）はシグネチャの定義のみです。
