# 運用・障害対応

> 動いているシステムを「壊れたとき気づく」「原因を特定する」「再発を防ぐ」ための技術をまとめています。

---

## はじめて読む人へ

運用・障害対応は、作ったシステムを動かし続けるための知識です。開発中は動いていても、本番ではアクセス増加、外部サービス障害、メモリ不足などが起きます。


### 読む前に押さえること

- ログは、システムで何が起きたかを後から追うための記録です。
- メトリクスは、CPU、メモリ、応答時間などを数値で見るために使います。
- ヘルスチェックは、サービスが生きているかを外から確認する仕組みです。

### 読み終えたら説明できること

- ログ、メトリクス、アラートの役割を説明できる。
- メモリリークやCPU高負荷の調査入口を理解できる。
- 障害時にまず何を確認するか判断できる。

---

## ログ

ログは、システムがいつ何をしたかを後から確認するための記録です。障害が起きたとき、ログがなければ原因を推測するしかありません。

良いログには、いつ、どの処理で、どのユーザーやリクエストに対して、何が起きたかが含まれます。逆に、ただ「エラー」とだけ出すログは、原因調査にはほとんど役立ちません。

ログは「何が起きたか」を記録するテキストです。障害発生時の唯一の手がかりになります。

### ログレベル

| レベル | 使いどき |
|--------|---------|
| DEBUG | 開発時の詳細情報（本番では出しません） |
| INFO | 正常な処理の記録（リクエスト受信・処理完了など） |
| WARNING | 問題にはなっていないが注意が必要な状態 |
| ERROR | 処理が失敗したが、システムは継続動作しています |
| CRITICAL | システム全体に影響する深刻な障害 |

### 良いログの書き方

```python
# 悪い例：何が起きたかわかりません
logger.error("Error occurred")

# 良い例：誰が・何を・どんな結果で
logger.error(
    "Failed to process order",
    extra={"user_id": user_id, "order_id": order_id, "reason": str(e)}
)
```

悪い例では、エラーが起きたことしか分かりません。どの注文で、どのユーザーに対して、なぜ失敗したのかをログから追えないため、障害時に再現や影響範囲の確認が難しくなります。

良い例では、メッセージに加えて `user_id`、`order_id`、`reason` を構造化して残しています。あとから「特定ユーザーの失敗だけを見る」「同じ reason のエラーを集計する」といった調査がしやすくなります。

**構造化ログ（JSON）：** ログを JSON 形式で出力すると、後から検索・集計しやすくなります。

```json
{"level": "ERROR", "message": "Failed to process order", "user_id": 42, "order_id": 101, "timestamp": "2024-01-15T10:30:00Z"}
```

JSONログは、ログ管理ツールでフィールドごとに検索できます。単なるテキストだと `"user_id": 42` を文字列検索するしかありませんが、構造化されていれば `user_id = 42` のような条件で絞り込めます。

### ログの保存と確認

| ツール | 用途 |
|--------|------|
| `journalctl` | systemd のサービスログを確認します |
| `docker logs <container>` | コンテナのログを確認します |
| CloudWatch Logs（AWS） | クラウドでのログ収集・検索 |
| Datadog / Grafana Loki | ログの集中管理・可視化 |

## メトリクス

メトリクスは「数値で表せる状態」の継続的な記録です。ログが「出来事」なら、メトリクスは「健康診断の数値」です。

### 監視すべき主要メトリクス

**システムレベル：**

| メトリクス | 異常の目安 | 意味 |
|-----------|-----------|------|
| CPU 使用率 | 常時 80% 超 | 処理能力の限界に近づいています |
| メモリ使用率 | 常時 90% 超 | スワップ発生のリスクがあります |
| ディスク使用率 | 90% 超 | ログがあふれてプロセスが止まります |
| ネットワーク I/O | 急激な増加 | DDoS・データ漏洩の可能性があります |

**アプリケーションレベル：**

| メトリクス | 意味 |
|-----------|------|
| リクエスト数（RPS） | トラフィックの量 |
| レイテンシ（P50/P95/P99） | 応答時間の分布 |
| エラーレート（5xx の割合） | サービスの健全性 |
| アクティブ接続数 | リソース消費の把握 |

**P99 レイテンシ：** 全リクエストの 99% が収まる応答時間です。平均だけ見ると「遅い 1% のユーザー」を見逃します。例えば平均が 50ms でも P99 が 5 秒なら、100 人に 1 人は 5 秒待たされています。大規模サービスでは 1% が数万人/日の体験に直結するため、P99 の監視が欠かせません。

## メモリリーク

プログラムが使い終わったメモリを解放せず、メモリ使用量が増え続ける問題です。

### 典型的なパターン（Python）

```python
cache = {}  # グローバルキャッシュ（上限なし）

def process(user_id: int):
    cache[user_id] = get_user_data(user_id)  # 追加するだけで削除しない
    ...
# ユーザーが増えるたびにメモリが増え続けます
```

この例では、一度アクセスしたユーザーデータが `cache` に追加され続けます。削除する処理や上限がないため、ユーザー数が増えるほどメモリ使用量も増えます。短時間のテストでは問題に見えなくても、本番で長時間動かすと顕在化します。

**解決策：**

```python
from functools import lru_cache

@lru_cache(maxsize=1000)  # 最大 1000 件でキャッシュを自動破棄します
def get_user(user_id: int):
    ...
```

`lru_cache(maxsize=1000)` は、最近使われた1000件までをキャッシュし、それ以上は古いものから捨てます。キャッシュは高速化に有効ですが、上限や有効期限を設けないとメモリリークの原因になります。

または TTL 付きのキャッシュライブラリを使います。

### 調査ツール

```bash
# プロセスのメモリ使用量を確認します
ps aux | grep python
top -p <PID>

# Python でのヒープ使用量確認
python -c "import tracemalloc; tracemalloc.start(); ..."
```

`ps` や `top` では、どのプロセスがメモリを使っているかを外側から見ます。`tracemalloc` はPythonプログラム内でどこがメモリを確保しているかを追うための道具です。外側の観察で異常を見つけ、内側のプロファイルで原因を絞る流れになります。

## CPU 高騰

CPU が常時高い状態が続く場合、以下を疑います。

| 原因 | 調査方法 |
|------|---------|
| 無限ループ・再帰 | スタックトレースを確認します |
| 非効率なアルゴリズム（O(n²) など） | プロファイラで特定します |
| GC（ガベージコレクション）の頻発 | メモリ使用量と合わせて確認します |
| 外部リクエストのタイムアウト待ち | スレッドが詰まってスレッド数が増加します |

```bash
# どのプロセスが CPU を使っているかを確認します
top
htop

# Python プロファイラ
python -m cProfile -s cumulative my_script.py
```

`top` や `htop` は、CPUを使っているプロセスを見つけるための入口です。特定のPythonスクリプトが重いと分かったら、`cProfile` で関数ごとの累積実行時間を見ます。どの関数が時間を使っているか分かると、アルゴリズム改善やキャッシュ導入の判断ができます。

## 可用性と冗長化

**可用性（Availability）：** システムが正常に稼働している時間の割合です。

| 可用性 | 年間ダウン時間 |
|--------|-------------|
| 99% | 約 3.65 日 |
| 99.9% | 約 8.77 時間 |
| 99.99%（フォーナイン） | 約 52 分 |
| 99.999%（ファイブナイン） | 約 5 分 |

### 単一障害点（SPOF）の排除

1 台が落ちると全体が止まるコンポーネントを「単一障害点（Single Point of Failure）」といいます。冗長化とはこれを排除することです。

!!! info ""
    単一障害点がある構成：
    クライアント → Web サーバー（1台）→ DB（1台）

    冗長化した構成：
    クライアント → LB → Web サーバー × 3 → DB（Primary + Replica）
単一障害点がある構成では、Webサーバー1台が落ちるだけでサービス全体が止まります。冗長化した構成では、ロードバランサが複数のWebサーバーに振り分け、DBもPrimaryとReplicaに分かれています。どこを冗長化するかは、どの障害を許容したいかによって決まります。

### フェイルオーバーとフェイルバック

- **フェイルオーバー：** 障害を検知して、バックアップに自動切り替えること
- **フェイルバック：** 障害が回復した後、元のプライマリに戻すこと

### ヘルスチェック

ロードバランサやオーケストレーター（Kubernetes など）は定期的にヘルスチェックエンドポイントを叩いて、応答しなくなったサーバーを自動で除外します。

```python
# FastAPI でのヘルスチェックエンドポイント
@app.get("/health")
def health():
    return {"status": "ok"}
```

ヘルスチェックは、ロードバランサや監視ツールが「このサーバーにリクエストを送ってよいか」を判断する入口です。単にプロセスが起動しているだけでなく、DB接続や依存サービスが使えるかまで確認する設計にすることもあります。

## 障害対応の流れ

!!! info ""
    1. 検知：アラートを受け取ります（監視ツール・ユーザー報告）
    2. 初動：影響範囲を把握します（何のサービスが・どれくらいの規模で影響しているか）
    3. 切り戻し：可能なら素早くロールバックまたは古いバージョンに戻します
    4. 原因調査：ログ・メトリクス・デプロイ履歴を確認します
    5. 修正：根本原因を修正してデプロイします
    6. 振り返り（ポストモーテム）：なぜ起きたか・再発防止策をドキュメント化します
障害対応では、最初から根本原因を完璧に当てにいくより、まず影響を止めることが優先です。直前のデプロイが怪しければ切り戻し、アクセス集中なら一時的に制限し、DBが落ちているならフェイルオーバーを確認します。その後、ログとメトリクスを見て原因を深掘りします。

**「誰が悪い」ではなく「なぜ起きたか」を追求することが重要です。** 責任追及より仕組みの改善が再発防止につながります（Blame-free Postmortem）。
