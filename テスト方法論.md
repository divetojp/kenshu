# テスト方法論

> コードが正しく動くことを確かめる方法を体系的に学びます。CI/CD と組み合わせることで「壊れたまま本番に出る」事故を防げます。

---

## はじめて読む人へ

テストは、プログラムが期待どおりに動くことを確認する仕組みです。手で毎回確認する代わりに、コードとして確認手順を残すことで、変更に強い開発ができます。


### 読む前に押さえること

- ユニットテストは、小さな部品を確認します。
- 統合テストは、複数の部品を組み合わせて確認します。
- E2E テストは、ユーザーに近い流れで確認します。

### 読み終えたら説明できること

- テストピラミッドの考え方を説明できる。
- pytest の基本的な書き方を読める。
- モックを使う理由を理解できる。

---

## なぜテストを書くのか

テストは、プログラムの使い方をコードとして残す役割も持ちます。関数にどんな入力を渡すと、どんな出力を期待するのかがテストに書かれていれば、あとから読む人は仕様を理解しやすくなります。

また、テストがあると安心して変更できます。新しい機能を足したり、内部構造を整理したりしたあとにテストを実行すれば、以前の動作を壊していないか確認できます。

```
「動くと思ってデプロイしたら壊れていた」
「直したつもりが別の場所を壊していた」
「半年後に自分のコードが何をしているかわからない」
```

このような問題は、コードを書いた本人の注意力だけでは防ぎきれません。テストは、期待する動作を自動で確認する仕組みとして、変更のたびに同じ確認を繰り返してくれます。

テストはこれらを防ぐ **安全網** です。

| テストがある場合 | テストがない場合 |
|---------------|---------------|
| 変更後に自動で確認できる | 手動確認に頼る（見落としやすい） |
| 壊れた場所がすぐわかる | 原因特定に時間がかかる |
| リファクタリングが安心してできる | 「動いているコードを触りたくない」になる |
| ドキュメントの役割も兼ねる | コードの使い方が不明確になる |

## テストの種類（テストピラミッド）

テストには粒度があります。小さい関数だけを見るテストは速く、大きなユーザー操作を通すテストは現実に近い代わりに遅くなります。

```
         /\
        /E2E\         （少ない・遅い・信頼性高い）
       /──────\
      /統合テスト\
     /──────────\
    / ユニットテスト \   （多い・速い）
   /────────────────\
```

テストピラミッドは、速いユニットテストを多く、遅い E2E テストを少なく持つのが基本だ、という考え方です。

### ユニットテスト（Unit Test）

**最小単位の関数・クラスが正しく動くか** を確認します。

- 速い（ミリ秒単位）
- 外部依存（DB・ネットワーク）を使わない
- 1 テスト = 1 つの振る舞いを確認

### 統合テスト（Integration Test）

**複数のコンポーネントが組み合わさったときに正しく動くか** を確認します。

- DB や API を実際に使う
- ユニットテストより遅い（秒単位）
- 「繋いだときに何かが壊れる」問題を発見できる

### E2E テスト（End-to-End Test）

**ユーザーの操作を模倣して、システム全体が正しく動くか** を確認します。

- ブラウザを実際に操作して確認（Playwright, Selenium）
- 最も遅い（分単位）
- 最も現実に近い検証

> **「ピラミッド型」にするのはなぜか：**  
> E2E テストは信頼性が高い反面、遅くて壊れやすいです。ユニットテストを多くしてE2E を少なくすることで、速くて安定したテストスイートを作れます。

## Python でのユニットテスト（pytest）

Python では、pytest を使うとシンプルにテストを書けます。まずテスト実行に必要なパッケージを入れます。

```bash
pip install pytest
```

### 基本的な書き方

テスト対象として、足し算と割り算の関数を用意します。割り算では、0 で割ろうとしたときに例外を出す仕様にしています。

```python
# calculator.py
def add(a, b):
    return a + b

def divide(a, b):
    if b == 0:
        raise ValueError("0 で割ることはできません")
    return a / b
```

次に、期待する動作を `test_` で始まる関数として書きます。pytest はこの名前の関数を自動で見つけて実行します。

```python
# test_calculator.py
from calculator import add, divide
import pytest

def test_add_positive_numbers():
    assert add(3, 5) == 8

def test_add_negative_numbers():
    assert add(-3, -5) == -8

def test_add_zero():
    assert add(0, 5) == 5

def test_divide_normal():
    assert divide(10, 2) == 5.0

def test_divide_by_zero():
    with pytest.raises(ValueError, match="0 で割ることはできません"):
        divide(10, 0)
```

`assert` は「この条件が成り立つはず」と確認する文です。`pytest.raises` は、指定した例外が発生することを確認します。

テストを書いたら、ターミナルで pytest を実行します。

```bash
pytest test_calculator.py        # テスト実行
pytest test_calculator.py -v     # 詳細表示
pytest                           # カレントディレクトリ以下を全部実行
```

出力：

```
test_calculator.py .....           [100%]
5 passed in 0.05s
```

`.` は 1 つのテストが通ったことを表します。失敗した場合は、どの `assert` が期待と違ったかが表示されます。

## テストの書き方：AAA パターン

良いテストは **Arrange → Act → Assert** の 3 段階で書きます。

AAA パターンを使うと、テストの読み方が安定します。まず準備し、次に対象の処理を実行し、最後に結果を確認します。

```python
def test_filter_passing_scores():
    # Arrange（準備）
    scores = [45, 82, 67, 91, 38]
    threshold = 70

    # Act（実行）
    result = [s for s in scores if s >= threshold]

    # Assert（確認）
    assert result == [82, 91]
```

テストが長くなったときも、この 3 つに分けて読むと、何を確認しているテストなのかが分かりやすくなります。

## モック（Mock）

テスト対象が **外部システムに依存している** とき、その依存を偽物に差し替えます。

外部 API、メール送信、現在時刻などは、テストのたびに実際に使うと遅く、不安定で、危険なことがあります。モックは、その外部依存を「呼ばれたことだけ確認できる偽物」に置き換える方法です。

```python
# email_service.py
import smtplib

def send_welcome_email(address):
    with smtplib.SMTP("smtp.example.com") as server:
        server.sendmail("noreply@example.com", address, "Welcome!")
    return True
```

このままだと実際にメールが送られてしまい、テストできません。

次のテストでは、`smtplib.SMTP` をモックに差し替えています。これにより、本物の SMTP サーバーへ接続せずに、送信処理が呼ばれたことだけ確認できます。

```python
# test_email_service.py
from unittest.mock import patch
from email_service import send_welcome_email

def test_send_welcome_email():
    with patch("smtplib.SMTP") as mock_smtp:
        # モック：SMTP サーバーへの接続を偽物に置き換える
        mock_server = mock_smtp.return_value.__enter__.return_value

        result = send_welcome_email("user@example.com")

        # メールが実際に送られた（sendmail が呼ばれた）か確認
        assert mock_server.sendmail.called
        assert result == True
```

`patch("smtplib.SMTP")` の中では、SMTP クラスが偽物に置き換わります。テスト対象の処理は通常通り呼びますが、外部への実通信は起きません。

**モックを使うべき対象：**

| 対象 | 理由 |
|------|------|
| 外部 API（天気、決済など） | テストのたびに課金・遅延が発生する |
| メール・SMS 送信 | 実際に送ってしまう |
| データベース（ユニットテストのみ） | 遅い・状態が残る |
| 現在時刻（`datetime.now()`） | テストの結果が時刻によって変わる |

> **モックの注意点：** モックを使いすぎると「テストは通るが本番では動かない」という状況になります。DB は統合テストで実際に使う判断も重要です（[CI/CD](CI-CD) 参照）。

## テスト駆動開発（TDD）

**コードを書く前にテストを書く** 開発スタイルです。

TDD は、先に「こう動いてほしい」というテストを書き、そのテストを通す最小限の実装を作り、最後に整理する流れです。

```
RED  → テストを書く（まだコードがないので失敗する）
GREEN → テストが通る最小限のコードを書く
REFACTOR → テストを通したまま、コードをきれいにする
```

**例：「FizzBuzz 関数を TDD で作る」**

まず、まだ存在しない `fizzbuzz` 関数に対して期待する動作を書きます。この時点ではテストは失敗します。

```python
# ステップ 1: テストを書く（まだ関数は存在しない）
def test_fizzbuzz_3():
    assert fizzbuzz(3) == "Fizz"

def test_fizzbuzz_5():
    assert fizzbuzz(5) == "Buzz"

def test_fizzbuzz_15():
    assert fizzbuzz(15) == "FizzBuzz"

def test_fizzbuzz_1():
    assert fizzbuzz(1) == "1"
```

次に、テストを通すための最小限の実装を書きます。

```python
# ステップ 2: テストが通る最小限の実装
def fizzbuzz(n):
    if n % 15 == 0: return "FizzBuzz"
    if n % 3 == 0:  return "Fizz"
    if n % 5 == 0:  return "Buzz"
    return str(n)
```

テストが通るか確認します。

```bash
pytest   # ✓ 全部通る
```

最後に、テストを通したままコードを整理します。テストがあるため、整理中に動作を壊していないか確認できます。

```python
# ステップ 3: リファクタリング（必要なら）
def fizzbuzz(n):
    result = ""
    if n % 3 == 0: result += "Fizz"
    if n % 5 == 0: result += "Buzz"
    return result or str(n)
```

**TDD の利点：**

- 関数の **インターフェース（何を受け取って何を返すか）** を先に考えられる
- 「何をテストすべきか」が明確になる
- 小さく作って確認するサイクルが速い

## パラメータ化テスト

同じロジックを複数の入力で確認するとき、テストを 1 つにまとめられます。

パラメータ化テストは、入力と期待値の組を表にして、同じテスト関数を何度も実行する仕組みです。似たテストを何個もコピーするより読みやすくなります。

```python
import pytest

@pytest.mark.parametrize("n, expected", [
    (1,  "1"),
    (3,  "Fizz"),
    (5,  "Buzz"),
    (15, "FizzBuzz"),
    (30, "FizzBuzz"),
])
def test_fizzbuzz(n, expected):
    assert fizzbuzz(n) == expected
```

この 1 つのテスト関数が、5 通りの入力で実行されます。失敗した場合は、どの入力ケースが失敗したかが表示されます。

```
test_fizzbuzz[1-1]      PASSED
test_fizzbuzz[3-Fizz]   PASSED
test_fizzbuzz[5-Buzz]   PASSED
test_fizzbuzz[15-FizzBuzz] PASSED
test_fizzbuzz[30-FizzBuzz] PASSED
```

## カバレッジ（Coverage）

テストがコードの何パーセントを実行しているかを計測します。

カバレッジは、テスト実行時にどの行が通ったかを測る指標です。テストされていない分岐や関数を見つける手がかりになります。

```bash
pip install pytest-cov
pytest --cov=calculator --cov-report=term-missing
```

出力：

```
Name             Stmts   Miss  Cover   Missing
----------------------------------------------
calculator.py       8      0   100%
```

> **カバレッジ 100% が目標ではありません。** 重要なのは「意味のある振る舞いをテストしているか」です。カバレッジは「テストされていないコードの発見」に使います。

## FastAPI でのテスト

FastAPI の API は、実際にサーバーを起動しなくても `TestClient` でテストできます。リクエストを送り、ステータスコードや JSON レスポンスを確認します。

```python
# main.py
from fastapi import FastAPI, HTTPException

app = FastAPI()

users = {
    1: {"user_id": 1, "name": "田中"},
}

@app.get("/users/{user_id}")
def get_user(user_id: int):
    if user_id not in users:
        raise HTTPException(status_code=404, detail="User not found")
    return users[user_id]
```

この `main.py` では、仮のユーザーデータを `users` という辞書で持っています。`/users/1` ならデータがあるので 200 を返し、存在しない ID なら `HTTPException` で 404 を返します。テストでは、正常系と異常系の両方を確認します。

次のテストでは、`/users/1` に GET リクエストを送り、期待する JSON が返ることを確認します。

```python
# test_main.py
from fastapi.testclient import TestClient
from main import app

client = TestClient(app)

def test_get_user():
    response = client.get("/users/1")
    assert response.status_code == 200
    assert response.json() == {"user_id": 1, "name": "田中"}

def test_get_user_not_found():
    response = client.get("/users/999")
    assert response.status_code == 404
```

2つ目のテストは、「存在しないユーザーを取得しようとしたら 404 を返す」という仕様を確認しています。テストはコードの現在の動きだけでなく、「この API はこう振る舞うべきだ」という約束を文章の代わりに表す役割も持ちます。

## CI でのテスト自動化

テストは **コードを push するたびに自動で実行** するのが理想です（[CI/CD](CI-CD) 参照）。

CI にテストを組み込むと、手元でテストを実行し忘れても、GitHub 上で自動確認されます。PR の段階で失敗に気づけるため、本番へ壊れた変更が入りにくくなります。

```yaml
# .github/workflows/test.yml
name: Test
on: [push, pull_request]
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-python@v4
        with: { python-version: '3.11' }
      - run: pip install -r requirements.txt
      - run: pytest --cov=src
```

このワークフローは、push と pull_request のたびに Ubuntu 環境を用意し、依存関係をインストールして pytest を実行します。

## まとめ

| 概念 | 要点 |
|------|------|
| テストピラミッド | ユニット（多い・速い）→ 統合 → E2E（少ない・遅い） |
| AAA パターン | Arrange → Act → Assert の 3 段階 |
| モック | 外部依存を偽物に置き換える |
| TDD | Red → Green → Refactor のサイクル |
| カバレッジ | テストされていないコードを発見する指標（100% が目的ではない） |
