ネットワーク基礎¶

インターネットとは¶

コンピューター同士がデータをやり取りするための世界規模のネットワーク です。

Web サイトを表示するとき、裏側では「あなたの PC」と「サーバー」の間でデータのやり取りが起きています。このページでは、その仕組みを理解します。

はじめて読む人へ¶

ネットワークは、コンピュータ同士が情報をやり取りする仕組みです。Webページを見る、APIを呼ぶ、GitHubへpushする、これらはすべてネットワーク通信の上に成り立っています。

読む前に押さえること¶

  • HTTP は、ブラウザとサーバーが会話するためのルールです。
  • DNS は、人間向けの名前をコンピュータ向けの住所に変換します。
  • ステータスコードは、通信結果を短い番号で伝えます。

読み終えたら説明できること¶

  • URL、HTTP、DNS の関係を説明できる。
  • GET と POST の違いを説明できる。
  • ステータスコードから問題の種類を推測できる。

IP アドレス¶

IP アドレス とは、インターネット上のコンピューターに割り当てられた 住所 のことです。

192.168.1.1
203.0.113.45

IPアドレスは、ネットワーク上で相手を特定するための番号です。192.168.x.x のようなアドレスは家庭や社内LANでよく使われるプライベートIPで、203.0.113.45 のようなアドレスはインターネット上の説明用例として使われます。

住所と同じで、どのコンピューターに届けるかを識別するために使います。

localhost（ローカルホスト）¶

localhost または 127.0.0.1 は、「自分自身の PC」 を指す特別なアドレスです。

npm run dev や docker compose up で開発サーバーを起動すると、http://localhost:4321 でアクセスできます。これは「自分の PC の中だけで動いているサーバー」で、インターネットには公開されていません。

ポート番号¶

ポート番号 とは、コンピューターの中の どのアプリに届けるか を示す番号です。

住所（IP アドレス）でビルに届け、部屋番号（ポート）で担当窓口を指定するイメージです。

ポート番号 用途
80 HTTP（通常の Web 通信）
443 HTTPS（暗号化された Web 通信）
4321 Astro 開発サーバー（デフォルト）
3000 Node.js 系フレームワークでよく使われます

http://localhost:4321 の :4321 がポート番号です。

ドメインと DNS¶

ドメインとは¶

diveto.jp のような、人間が読めるサイトの住所です。コンピューターは IP アドレスで通信しますが、数字の羅列は覚えにくいため、ドメインが使われます。

DNS（Domain Name System）とは¶

ドメインを IP アドレスに変換する仕組みです。インターネット上の「電話帳」と例えられます。

ブラウザ → DNS に問い合わせ「diveto.jp の IP は？」 ↓ DNS が回答「203.0.113.45 です」 ↓ ブラウザ → 203.0.113.45 のサーバーにアクセスします Cloudflare はこの DNS の管理もしてくれるサービスです。

ブラウザにURLを入力すると、いきなりWebサーバーへ行くのではなく、まずDNSで「このドメインはどのIPアドレスか」を調べます。DNSの結果は一定時間キャッシュされるため、設定変更がすぐ反映されないことがあります。

HTTP と HTTPS¶

HTTP は、クライアントとサーバーがリクエストとレスポンスをやり取りするためのルールです。ブラウザでページを開くと、ブラウザがサーバーへリクエストを送り、サーバーがHTMLやJSONなどを返します。

HTTPS は、このHTTP通信を暗号化したものです。途中で通信を盗み見られたり、改ざんされたりするリスクを下げるため、公開サイトではHTTPSが基本になります。

情報工学メモ：TCP/IP とは（信頼性のある通信の仕組み）、パケットとは
インターネットの通信は、データを小さな塊に分割した パケット という単位でやり取りします。各パケットは別々の経路を通って相手に届くことがあります。IP（Internet Protocol） はパケットを宛先に届ける「住所配送」の役割を、TCP（Transmission Control Protocol） はパケットが全部届いたかを確認して「信頼性のある通信」を実現する役割を担います。TCP は接続開始時に「SYN を送る→SYN-ACK が返る→ACK を送る」という 3 段階（3 ウェイハンドシェイク）で接続を確立し、届かなかったパケットは再送します。HTTP は TCP/IP の上で動くプロトコルです。

HTTP（HyperText Transfer Protocol）¶

ブラウザとサーバーがデータをやり取りするときのルール（プロトコル） です。

http:// で始まる URL がこれにあたります。

HTTPS（HTTP Secure）¶

情報工学メモ：TLS/SSL の仕組み（対称暗号・非対称暗号・証明書）
HTTPS は TLS （旧称 SSL）という仕組みで通信を暗号化します。暗号化には 2 種類あります。非対称暗号（公開鍵暗号）は公開鍵で暗号化し秘密鍵でしか復号できない方式で、鍵の安全な交換に使います（RSA など）。対称暗号 は同じ鍵で暗号化・復号する方式で高速です（AES など）。TLS ではまず非対称暗号で共通鍵を安全に交換し、その後の通信は高速な対称暗号で行います。また サーバー証明書 を使って「このサーバーが本物かどうか」を第三者（認証局）が保証する仕組みも含まれ、偽サイトへの誘導（フィッシング）を防ぎます。

HTTP を 暗号化 したものです。https:// で始まります。現代の Web サイトはほぼすべて HTTPS を使います。

HTTP HTTPS
暗号化 なし あり
安全性 低い（盗聴リスク） 高い
ブラウザ表示 ⚠️ 保護されていない通信 🔒 鍵マーク

HTTP と HTTPS の違いは、URLの見た目だけではありません。HTTPでは通信内容が平文に近い形で流れるため、同じネットワーク上の第三者に盗み見られるリスクがあります。HTTPSではTLSによって暗号化され、接続先のサーバーが本物かも証明書で確認されます。

HTTP メソッド¶

HTTP でサーバーに「何をしたいか」を伝えるための種類です。主に 4 種類あります。

メソッド 意味 用途の例
GET データを取得します ページを表示する、記事一覧を取得する
POST データを作成します フォームを送信する、新規登録する
PUT / PATCH データを更新します プロフィールを変更する
DELETE データを削除します 記事を削除する

ブラウザでページを開くと GET が使われます。フォームの「送信」ボタンを押すと POST が使われます。

HTTPメソッドは、Web API の設計にもそのまま使われます。記事一覧を見るなら GET /articles、新しい記事を作るなら POST /articles、記事を削除するなら DELETE /articles/1 のように、操作の意味をメソッドで表します。

HTTP ステータスコード¶

サーバーがリクエストに対して返す「結果の番号」です。3 桁で表され、最初の数字で大まかな意味が分かります。

よく見るステータスコード¶

コード 意味 説明
200 OK 成功 リクエストが正常に処理されました
301 Moved Permanently リダイレクト URL が永久に移動しました（旧 URL → 新 URL）
304 Not Modified キャッシュ有効 キャッシュのデータをそのまま使ってよいです
400 Bad Request リクエスト不正 送ったデータに問題があります
401 Unauthorized 認証が必要 ログインが必要なページにアクセスしました
403 Forbidden アクセス禁止 権限がないリソースにアクセスしました
404 Not Found 見つからない 指定した URL にリソースが存在しません
500 Internal Server Error サーバーエラー サーバー側で問題が発生しました
503 Service Unavailable 利用不可 サーバーが過負荷またはメンテナンス中です

開発者ツールで確認する¶

ブラウザの開発者ツール（F12）の Network タブ を開くと、各リクエストのステータスコードを確認できます。404 や 500 が出ていたらエラーの原因を調査しましょう。

ステータスコードを見ると、問題が自分のリクエスト側にあるのか、サーバー側にあるのかを大まかに切り分けられます。4xx はURLや権限、送信データなどクライアント側の問題であることが多く、5xx はサーバー内部のエラーや過負荷を示します。

リクエストとレスポンスの流れ¶

ブラウザが Web ページを表示するまでの流れを順番に見てみましょう。

① ブラウザに URL を入力します ↓ ② DNS でドメインを IP アドレスに変換します ↓ ③ サーバーに「このページをください」（HTTP リクエスト） ↓ ④ サーバーが HTML・CSS・画像などを返します（HTTP レスポンス） ↓ ⑤ ブラウザが受け取ったデータを解析・表示します この流れでは、DNS、HTTP、HTML/CSS/画像の取得、ブラウザでの描画が連続して起きています。ページが遅いときは、DNSが遅いのか、サーバーの応答が遅いのか、画像が大きいのか、JavaScriptの実行が重いのかを分けて考えます。

開発者ツールで確認する¶

ブラウザの開発者ツール（F12 キーで開きます）の Network タブ を見ると、どのファイルが何ミリ秒で読み込まれたか確認できます。

CDN とキャッシュ¶

CDN（Content Delivery Network）とは¶

世界中にサーバーを分散配置して、ユーザーに近いサーバーからコンテンツを届ける仕組み です。

日本のユーザーが、アメリカにあるサーバーのサイトにアクセスする場合：

CDN なし：日本 → アメリカのサーバー（遅い） CDN あり：日本 → 日本国内の CDN サーバー（速い） Cloudflare は世界最大級の CDN サービスです。

CDN は、HTML、CSS、画像、JavaScriptのような静的ファイルをユーザーに近いサーバーから配ります。物理的な距離が短くなると通信の往復時間が短くなり、ページ表示が速くなります。

キャッシュとは¶

一度取得したデータを一時的に保存しておき、次回以降は再取得せずに使い回す仕組みです。

初回アクセス：サーバーからデータを取得します（時間がかかります） 2 回目以降 ：キャッシュから取得します（速い）

キャッシュが原因でサイトの更新が反映されない場合
ブラウザでハードリフレッシュ（Win: Ctrl + Shift + R、Mac: Command + Shift + R）を試してください。

キャッシュは高速化に有効ですが、古いファイルが残る原因にもなります。開発中にCSSを直したのに画面が変わらない場合、コードではなくブラウザやCDNのキャッシュを見ている可能性があります。

hp に関係するネットワークの流れ¶

あなたが hp を更新するとき

① git push → GitHub にコードが届きます ↓ ② Cloudflare Pages が変更を検知します ↓ ③ Cloudflare がビルドして HTML を生成します ↓ ④ Cloudflare の CDN サーバー（世界各地）に配信します ↓ ⑤ ユーザーが diveto.jp にアクセスします ↓ ⑥ DNS が IP アドレスを返します → Cloudflare サーバーに接続します ↓ ⑦ 最寄りの CDN サーバーから HTML・CSS を受け取ります ↓ ⑧ ブラウザが表示します この流れを見ると、サイト公開は「GitHubに置いたファイルをユーザーが直接見る」わけではないことが分かります。GitHubはソースコードの置き場、Cloudflare Pagesはビルドと配信の担当、DNSはドメイン名を配信先へ向ける担当です。

よくある疑問¶

Q. http:// と https:// は何が違う？
A. HTTPS は通信内容が暗号化されています。パスワードや個人情報を扱うサイトは必ず HTTPS を使います。Cloudflare は自動的に HTTPS を有効にしてくれます。

Q. DNS の設定を変えたのにサイトが切り替わらない
A. DNS の変更は世界中に伝播するまで最大 72 時間かかる場合があります（通常は数時間以内）。

Q. ポート番号はいつも指定しないといけない？
A. HTTP（80 番）と HTTPS（443 番）は省略できます。http://example.com は実際には http://example.com:80 です。
