# Docker

## Docker とは

**アプリケーションとその実行環境をまるごとパッケージ化して、どこでも同じように動かせるようにするツール** です。

---

## はじめて読む人へ

Docker は、アプリを動かす環境を箱ごとそろえるための道具です。自分のPCでは動くのに他の人のPCでは動かない、という問題を減らします。


### 読む前に押さえること

- イメージはコンテナの設計図、コンテナは実際に動く箱です。
- Dockerfile は、環境を作る手順を書いたファイルです。
- docker compose は、複数のコンテナをまとめて起動するために使います。

### 読み終えたら説明できること

- イメージとコンテナの違いを説明できる。
- `docker compose up` が何をしているか理解できる。
- コンテナを起動・停止・再ビルドできる。

---

## なぜ Docker が必要か

> **情報工学メモ：仮想マシン（VM）とコンテナの違い、カーネルとは何か**  
> **カーネル** とは OS の核心部分で、CPU・メモリ・ディスクなどのハードウェアをプログラムから利用できるよう仲介する役割を担います。**仮想マシン（VM）** は、カーネルを含む OS 全体をソフトウェアで丸ごとエミュレートするため、起動が遅くメモリ消費が大きくなります。一方 **Docker コンテナ** は、ホスト OS（Mac や Windows）のカーネルを共有しつつ、ファイルシステムやプロセス空間だけを分離します。OS 全体を複製しないため、起動が数秒と速く、容量も小さい点が特徴です。

プログラムを動かすには「環境」が必要です。たとえば Node.js のバージョン、インストールされているパッケージ、OS の設定など、様々なものが揃っていないと動きません。

Docker を使わない場合、こんな問題が起きます：

> 「自分の PC では動いたのに、別のメンバーの PC では動かない…」  
> 「先輩の環境構築手順が長すぎて、どこかでミスった…」

Docker を使うと：

次の図は、Docker を使ったときに同じ環境が複数の場所で再現されるイメージです。重要なのは、アプリだけでなく、実行に必要な環境も一緒に扱う点です。

!!! info ""
    ```
    Aさんの Windows でも
    Bさんの Mac でも
    本番サーバーの Linux でも
    → まったく同じ環境で動きます
    ```

Docker は「この環境で動かしてください」という手順をイメージとして固定します。そのイメージから起動したコンテナは、どのマシンでも同じ依存関係を持つため、環境差によるトラブルを減らせます。

## 基本的な用語

Docker を理解するときは、イメージとコンテナを分けて考えます。イメージは実行環境の設計図で、コンテナはその設計図から実際に起動したプロセスです。

同じイメージから複数のコンテナを起動できます。つまり、環境の作り方をイメージとして固定しておくことで、誰のPCでも同じようにアプリを動かしやすくなります。

> **情報工学メモ：レイヤー構造とファイルシステムの仕組み**  
> Docker イメージは **レイヤー（層）** の積み重ねで構成されています。「OS の基盤」「Node.js のインストール」「アプリのコピー」など、Dockerfile の命令 1 行ごとに 1 つのレイヤーが作られます。一度作ったレイヤーはキャッシュされるため、変更のある部分から下だけを再ビルドすれば済み、高速です。コンテナを起動すると、読み取り専用のイメージのレイヤー群の上に、書き込み可能な薄いレイヤーが追加されます。コンテナを削除するとその書き込みレイヤーだけが消え、イメージ本体は残ります。

最初はイメージとコンテナだけ覚えておけば大丈夫です。

| 用語 | 意味 | 例え |
|------|------|------|
| **イメージ** | 環境の設計図（テンプレート） | 料理のレシピ |
| **コンテナ** | イメージから起動した実際の環境 | レシピ通りに作った料理 |
| **Dockerfile** | イメージを作る手順を書いたファイル | レシピの書き方 |
| **Docker Compose** | 複数のコンテナをまとめて管理するツール | コース料理のセット |

## このプロジェクトでの使い方

### 開発環境を起動する

> リポジトリのクローン（初回のダウンロード）は [環境構築](環境構築) ページで済んでいるはずです。  
> `ota_hp` フォルダに移動してから、以下のコマンドを実行してください。

コンテナを起動する前に、まずプロジェクトのフォルダへ移動します。`docker compose up` は、そのフォルダにある `docker-compose.yml` を読んで、必要なコンテナを起動します。

```bash
# ota_hp フォルダに移動します
cd ota_hp

# コンテナを起動します
docker compose up
```

このコマンドを実行すると、Docker は設定ファイルを読み、必要なイメージを取得またはビルドし、コンテナを起動します。ログがターミナルに流れ続けるのは、開発サーバーがその中で動いているからです。

ターミナルに `http://localhost:4321` と表示されたら、ブラウザでそのアドレスを開いてください。

> **初回は時間がかかることがあります。**  
> Docker がイメージをダウンロードしているためです。2 回目以降は速くなります。

### 停止する

`docker compose up` を前面で実行している場合は、ターミナルで `Ctrl + C` を押すと停止できます。別のターミナルから明示的に止める場合は `docker compose down` を使います。

```bash
# ターミナルで Ctrl + C を押します
# または別のターミナルウィンドウで
docker compose down
```

`down` はコンテナを停止し、Compose が作ったコンテナを削除します。ただし、ローカルのソースコードは消えません。

### バックグラウンドで起動する

ターミナルを占有したくない場合は `-d` オプションをつけます。

バックグラウンド起動では、ログが画面に流れ続けません。別の作業をしながらサーバーを起動しておきたいときに便利です。

```bash
docker compose up -d
# → ターミナルを使いながら開発できます

docker compose down
# → 止めるときはこちら
```

バックグラウンドで動かした場合、エラーやアクセスログを見たいときは `docker compose logs -f` を使います。

## よく使うコマンド

Docker のコマンドは、最初は「起動」「停止」「ログ確認」「再起動」の 4 種類を覚えれば十分です。Compose を使う場合は、複数コンテナをまとめて操作できます。

```bash
docker compose up          # コンテナを起動します（ログをターミナルに表示します）
docker compose up -d       # バックグラウンドで起動します
docker compose down        # コンテナを停止・削除します
docker compose logs -f     # ログをリアルタイムで確認します（-d で起動中に使います）
docker compose restart     # コンテナを再起動します
```

`logs -f` の `-f` は follow の意味で、新しいログを追いかけて表示します。画面が更新されない、API が返らない、といったときはまずログを見る習慣をつけます。

### コンテナの中に入る

何かのコマンドをコンテナ内で直接実行したい場合：

コンテナの中に入ると、その環境でシェルを操作できます。依存パッケージが入っているか、ファイルがコピーされているか、コマンドが動くかを確認するのに使います。

```bash
docker compose exec web sh
```

`/app #` のような表示になったらコンテナの中に入れています。  
`exit` で抜けられます。

## docker-compose.yml の読み方

プロジェクトのルートにある `docker-compose.yml` ファイルが、コンテナの設定を定義しています。

Compose ファイルは、コンテナの設定表です。どのイメージを使うか、どのフォルダを共有するか、どのポートを外に出すか、起動時に何を実行するかを書きます。

```yaml
services:
  web:                         # ← サービス（コンテナ）の名前
    image: node:20-alpine      # ← 使用するイメージ（Node.js 20 の軽量版）
    working_dir: /app          # ← コンテナ内の作業ディレクトリ
    volumes:
      - .:/app                 # ← ローカルのファイルをコンテナと共有します
    ports:
      - "4321:4321"            # ← ホスト:コンテナ のポートマッピング
    command: npm run dev       # ← 起動時に実行するコマンド
```

`services` の下に、起動したいコンテナをサービスとして定義します。この例では `web` というサービスが 1 つあり、Node.js のイメージを使って開発サーバーを起動します。

`ports` の `"4321:4321"` は、左が自分の PC のポート、右がコンテナ内のポートです。ブラウザで `localhost:4321` を開くと、コンテナ内の 4321 番で動くサーバーへつながります。

> **volumes の意味：**  
> ローカルでファイルを編集すると、コンテナの中にも即座に反映されます。  
> これによって、コンテナを再起動しなくても変更がブラウザに反映されます。

## トラブルシューティング

### ポートが使用中というエラーが出る

ポートは、PC 上で動いているアプリへの入口番号です。同じ番号を複数のアプリが同時に使うことはできません。

```
Error: listen EADDRINUSE: address already in use :::4321
```

→ 別のプロセスが同じポートを使っています。  
まず `docker compose down` してから、もう一度 `docker compose up` してみてください。

### 変更がブラウザに反映されない

→ `docker compose restart` を試してください。  
それでも解決しない場合は `docker compose down` してから `docker compose up` で再起動します。

### パッケージを追加した場合

`package.json` に変更があった場合は、コンテナを再ビルドする必要があります。

依存パッケージが変わったときは、既存のコンテナを起動し直すだけでは足りないことがあります。イメージを作り直して、新しい依存関係をコンテナへ入れ直します。

```bash
docker compose down
docker compose up --build
```

`--build` を付けると、必要に応じてイメージを再ビルドしてから起動します。パッケージを追加したのにコンテナ内で見つからない場合は、この操作を試します。

## Docker をインストールする

まだインストールしていない場合は **Docker Desktop** を使うのが簡単です。

1. [Docker Desktop の公式サイト](https://www.docker.com/products/docker-desktop/) にアクセスします
2. 自分の OS（Windows / Mac）に合ったものをダウンロードします
3. インストールして起動します
4. インストール確認：

インストール後は、ターミナルで Docker と Compose のバージョンが表示されるか確認します。バージョンが出れば、コマンドが使える状態です。

```bash
docker --version
docker compose version
```

バージョンが表示されれば OK です。

## よくある疑問

**Q. Docker を使わずに直接 `npm run dev` ではダメ？**  
A. Node.js が PC にインストールされていれば動きます。ただし、Node.js のバージョンや環境の違いで動かないことがあるため、チームでは Docker を使うことを推奨しています。

**Q. コンテナを消したらファイルも消える？**  
A. `volumes` の設定により、ローカルのファイルとコンテナが同期されています。コンテナを削除してもローカルのファイルは消えません。
