# セキュリティ

## なぜ Web セキュリティが重要か

Web サイトを公開するということは、世界中の人からアクセスできる状態にすることです。悪意のある人間がサイトを攻撃したり、情報を盗もうとすることがあります。

「自分たちのサイトは小さいから大丈夫」は危険な考えです。多くの攻撃は自動化されており、規模に関係なく標的になります。

---

## はじめて読む人へ

セキュリティは、攻撃者だけを意識するものではありません。うっかり秘密情報を公開しない、不要な権限を渡さない、依存パッケージを放置しない、といった日常の習慣が基本になります。

コードやコマンドが出てきたら、最初から全部を覚えようとしなくて大丈夫です。まずは「何を入力し、何が処理され、何が出力されるのか」を文章で説明できるように読むと、手を動かす前の理解が安定します。

### 読む前に押さえること

- 公開されたWebサイトは、自動化された攻撃の対象になります。
- 秘密情報はコードではなく環境変数やSecretsで管理します。
- 最小権限にすると、ミスや攻撃の被害範囲を小さくできます。

### 読み終えたら説明できること

- HTTPS、秘密情報、権限管理の基本を説明できる。
- `.env` を Git に含めてはいけない理由を説明できる。
- XSS や CSRF の大まかな危険性を理解できる。

## HTTPS（暗号化通信）

> **情報工学メモ：暗号化の仕組み（対称暗号・非対称暗号の違い、RSA・AES）**  
> 暗号化には大きく 2 種類あります。**対称暗号** は暗号化と復号に同じ鍵を使う方式で、処理が高速です（代表例：**AES**）。ただし「鍵をどうやって安全に相手に渡すか」という問題があります。**非対称暗号**（公開鍵暗号）は「公開鍵で暗号化し、対応する秘密鍵でしか復号できない」方式で、公開鍵は誰に渡しても安全です（代表例：**RSA**）。HTTPS（TLS）ではまず非対称暗号を使って共通鍵を安全に交換し、その後の通信は高速な対称暗号（AES）で行うというハイブリッド方式を採用しています。

サイトを HTTPS で提供することが、最も基本的なセキュリティ対策です。

> HTTPS の仕組みは [ネットワーク基礎](ネットワーク基礎) ページで解説しています。

Cloudflare Pages にデプロイすると、**自動的に HTTPS が有効になります**。追加の設定は不要です。

ブラウザのアドレスバーに 🔒 マークが表示されていれば OK です。

## 秘密情報の管理（最重要）

秘密情報とは、知らない人に見られると悪用される値のことです。API キー、DB パスワード、アクセストークンなどは、コードそのものとは違い、外へ漏れると実際のサービスやデータにアクセスされる危険があります。

重要なのは、秘密情報を「あとで消せばよい」と考えないことです。GitHub に一度 push すると履歴に残るため、削除コミットだけでは不十分です。漏れたキーは無効化して作り直します。

### 絶対に Git にコミットしてはいけないもの

以下の情報をコードに直接書いて GitHub にプッシュすると、世界中から見られてしまいます。

- API キー・シークレットキー
- パスワード
- データベースの接続情報
- アクセストークン

### .env ファイルを使う

秘密情報は **`.env`** ファイルに書き、`.gitignore` で Git 管理から除外します。

`.env` は、ローカル開発用の環境変数を書くファイルです。API キーや DB 接続情報のように、環境ごとに違う値や外に見せてはいけない値を入れます。

```bash
# .env ファイル（Git には含めません）
API_KEY=your_secret_key_here
DATABASE_URL=postgresql://user:password@localhost/db
```

この例では、`API_KEY` と `DATABASE_URL` が秘密情報です。値そのものは自分の PC にだけ置き、GitHub へ push しません。

```bash
# .gitignore に追記します（このファイルは Git に含まれます）
.env
.env.local
```

`.gitignore` に書くことで、Git は `.env` を変更ファイルとして扱わなくなります。秘密情報を守るには、`.env` を作るだけでなく、Git に含まれない設定まで確認する必要があります。

> **`.env` ファイルが既に GitHub にプッシュされてしまった場合**  
> そのキー・パスワードは即座に無効化・再発行してください。Git の履歴から削除するだけでは不十分です（履歴は残るため）。

### GitHub Secrets（CI/CD での秘密情報）

GitHub Actions や Cloudflare Pages で環境変数を使う場合、**GitHub Secrets** または **Cloudflare の環境変数設定** を使います。

**Cloudflare Pages での設定：**
1. ダッシュボード → Pages プロジェクト → **Settings** → **Environment variables**
2. 変数名と値を入力して保存します

コードには変数名だけを書きます。

コード側では、秘密情報の実際の値ではなく、環境変数名を参照します。実際の値は GitHub や Cloudflare の設定画面に保存します。

```js
const apiKey = process.env.API_KEY;  // 実際の値はコードに書きません
```

この形にすると、開発環境・プレビュー環境・本番環境で別々の API キーを使えます。漏洩時も、該当する環境のキーだけを無効化できます。

## よくある Web の脆弱性

### XSS（クロスサイトスクリプティング）

悪意のある JavaScript をページに埋め込む攻撃です。

**悪い例（ユーザー入力をそのまま表示）：**

次のようにユーザー入力をそのまま HTML として扱うと、入力に悪意あるスクリプトが含まれていた場合に実行される危険があります。

```html
<!-- ユーザーが入力した値を直接 HTML に埋め込む → 危険 -->
<p>{userInput}</p>
```

**Astro でのケア：**  
Astro はデフォルトで `{変数}` の内容を **エスケープ**（無害化）してくれます。HTML をそのまま埋め込む `set:html` を使うときだけ注意が必要です。

> **エスケープとは：** `<script>` のような HTML の特殊文字を `&lt;script&gt;` のような無害な文字列に変換することです。エスケープされた文字はブラウザが「コード」ではなく「テキスト」として表示するため、JavaScript として実行されません。

### CSRF（クロスサイトリクエストフォージェリ）

ユーザーが意図しない操作をさせる攻撃です。フォームを持つサイトで問題になります。静的サイトであれば基本的に心配不要です。

### 依存パッケージの脆弱性

使用しているパッケージに既知の脆弱性がある場合があります。

依存パッケージは便利ですが、そこに脆弱性が見つかることもあります。`npm audit` は、現在の依存関係に既知の脆弱性がないかを確認するコマンドです。

```bash
# 脆弱性を確認します
npm audit

# 自動修正を試みます
npm audit fix
```

`npm audit fix` は、互換性を保てる範囲で安全なバージョンへ更新を試みます。実行後は、サイトが壊れていないかビルドや動作確認を行います。

### npm audit の出力を読む

監査結果には、脆弱性の件数と深刻度が表示されます。深刻度によって対応の優先順位を決めます。

```
found 5 vulnerabilities (1 low, 2 moderate, 2 critical)
```

| 深刻度 | 意味 | 対応方針 |
|--------|------|---------|
| **critical** | 即座に対応が必要 | 直ちに `npm audit fix` を実行します |
| **high** | できるだけ早く対応 | 今週中に対応します |
| **moderate** | 影響は限定的 | 次のリリースまでに対応します |
| **low** | ほぼ影響なし | 余裕があれば対応します |

詳細を見るには：

すべての低リスク警告まで見ると情報量が多すぎる場合があります。一定以上の深刻度だけを表示して、優先度の高い問題から確認できます。

```bash
npm audit --audit-level=moderate   # moderate 以上だけ表示します
```

### Dependabot（GitHub の自動脆弱性検出）

GitHub は自動的にリポジトリの依存パッケージを監視し、脆弱性を発見すると **Dependabot** が自動で PR を作成してくれます。

**通知が来たら：**
1. PR の内容を確認します（どのパッケージの何が問題か）
2. `npm audit fix` でローカルも更新します
3. テスト・確認後にマージします

## アクセス権限の管理

### GitHub の権限設定

| ロール | できること |
|--------|-----------|
| **Owner** | すべての操作。リポジトリ削除も可能 |
| **Admin** | 設定変更・メンバー管理 |
| **Write** | コードの push・マージ |
| **Read** | コードの閲覧のみ |

**最小権限の原則** ：必要最小限の権限だけを付与します。関係のない人に Admin 権限を与えないようにしましょう。

### 不要になったアクセスの削除

メンバーが抜けた場合は、速やかにリポジトリとその他サービスへのアクセスを削除します。

- GitHub: Settings → Collaborators → Remove
- Cloudflare: アカウントメンバーから削除

## パスワード・アカウント管理

### 強力なパスワードを使う

- 12 文字以上
- 英大文字・小文字・数字・記号を含む
- サービスごとに異なるパスワードを使う

**パスワードマネージャーの使用を強く推奨します** （1Password, Bitwarden など）。

### 二要素認証（2FA）を有効にする

GitHub, Cloudflare などの重要なサービスでは必ず二要素認証を設定します。

**GitHub での設定：**
Settings → Password and authentication → Two-factor authentication

## セキュリティチェックリスト

プロジェクトを公開する前に確認しましょう。

チェックリストは、セキュリティ対策を「知っている」だけで終わらせず、公開前に実際に確認するためのものです。

- [ ] サイトが HTTPS で提供されている（Cloudflare の 🔒 マーク確認）
- [ ] `.env` ファイルが `.gitignore` に追加されている
- [ ] API キーなどの秘密情報がコードに直接書かれていない
- [ ] GitHub リポジトリのアクセス権限が適切に設定されている
- [ ] `npm audit` でパッケージの脆弱性がないか確認した
- [ ] メンバー全員が二要素認証を設定している

## よくある疑問

**Q. 静的サイトでもセキュリティは必要？**  
A. はい。サーバーサイドの攻撃リスクは少ないですが、依存パッケージの脆弱性、秘密情報の漏洩、権限管理の問題は静的サイトでも起きます。

**Q. GitHub のリポジトリを Public にしていいの？**  
A. コード自体は問題ありませんが、`.env` ファイルや API キーが含まれていないことを必ず確認してください。

**Q. `npm audit` で脆弱性が出たけど修正できない**  
A. `npm audit fix --force` で強制修正できますが、依存関係が壊れることがあります。まず `npm audit fix` を試し、解決しない場合はチームメンバーに相談しましょう。
