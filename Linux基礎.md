# Linux 基礎

## Linux とは

**サーバーで最もよく使われる OS（オペレーティングシステム）** です。

あなたが普段使う PC は Windows や macOS ですが、Web サービスのサーバー・クラウド環境はほぼ Linux で動いています。Docker のコンテナも中身は Linux です。

> **Windows ユーザーへ：**  
> Windows のコマンドプロンプト・PowerShell とは文法が異なります。Docker を使う場合はコンテナ内が Linux 環境になるため、このページのコマンドはコンテナ内でそのまま使えます。ローカルで Linux コマンドを試したい場合は **WSL 2**（Windows Subsystem for Linux）が便利です。

> **Mac ユーザーへ：**  
> macOS は Linux と同じ UNIX 系 OS のため、ターミナルコマンドの多くが共通です。このページの内容はほぼそのまま Mac でも使えます。

## はじめて読む人へ

Linux は、サーバーやコンテナの中でよく使われる OS です。Web アプリや機械学習の本番環境は Linux 上で動くことが多いため、最低限の操作を知っているとトラブル対応がしやすくなります。


### 読む前に押さえること

- ファイル権限は「誰が何をできるか」を表します。
- プロセスは、いま実行中のプログラムです。
- ログや設定ファイルを読めることは、開発と運用の基本です。

### 読み終えたら説明できること

- 権限、プロセス、ディスク使用量を確認できる。
- SSH が何のための仕組みか説明できる。
- 簡単なシェルスクリプトの流れを読める。

## ファイルシステムの構造

Linux では、すべてのファイルやフォルダが `/` を頂点とする1本の木構造の中に配置されます。Windows のように C ドライブ、D ドライブと分かれている感覚とは少し違います。

サーバー上で作業するときは、自分が今どこにいるかを常に意識します。設定ファイル、ログ、アプリのコード、秘密情報は置かれる場所が違うため、パスを読めることがトラブル対応の第一歩になります。

Linux のファイル構造は Windows と異なります。すべてが `/`（ルート）から始まります。

次の図は、Linux の代表的なディレクトリ構造です。サーバー作業では、設定は `/etc`、ログは `/var/log`、ユーザーの作業場所は `/home/ユーザー名` にあることが多いです。

!!! info ""
    ```text
    /                    ← ルート（最上位）
    ├── home/            ← ユーザーのホームディレクトリ
    │   └── yamada/      ← ユーザー yamada のホーム（~ と同じ）
    ├── etc/             ← 設定ファイル
    ├── var/             ← ログ・データなど変動するファイル
    │   └── log/         ← ログファイル
    ├── usr/             ← アプリケーション・コマンド
    ├── tmp/             ← 一時ファイル（再起動で消えます）
    └── root/            ← root ユーザーのホーム
    ```
パスが `/` から始まる場合は絶対パス、今いる場所から見た位置を書く場合は相対パスです。エラーログや設定ファイルを探すときは、まず絶対パスを読めるようになることが大切です。

## ファイル権限

> **情報工学メモ：Unix のユーザー・グループモデルとは**  
> Linux（Unix 系 OS）では、すべてのファイルに「所有者（owner）」「グループ（group）」「その他（others）」の 3 層の権限が設定されています。1 台のコンピューターを複数人で共有することを前提に設計されたため、ユーザーごとにアクセスできるファイルを細かく制御できます。`root`（スーパーユーザー）はすべての権限を持つ特別なユーザーで、OS の設定変更などに使います。`sudo` コマンドは「一時的に root として実行する」という意味です。

Linux ではファイルやフォルダに **誰が何をできるか** の権限が設定されています。

権限を理解するには、まず `ls -l` でファイル一覧を詳しく表示します。通常の `ls` よりも、所有者、グループ、サイズ、更新日時、権限が見えるようになります。

```bash
ls -l
```

たとえば、次のような出力が表示されます。左端の記号列が権限を表しています。

```
-rw-r--r-- 1 yamada staff  1234 Mar  1 10:00 README.md
drwxr-xr-x 3 yamada staff    96 Mar  1 10:00 src/
```

### 権限の読み方

権限の 10 文字は、ファイル種別、所有者、グループ、その他のユーザーの権限に分かれます。最初は暗号のように見えますが、3 文字ずつに分けると読めます。

!!! info ""
    ```text
    -rw-r--r--
    │ │││ │││ │└ others（その他のユーザー）: r-- = 読み取りのみ
    │ │││ └┘└ group（グループ）: r-- = 読み取りのみ
    │ └┘└ owner（所有者）: rw- = 読み取り・書き込み
    └ ファイル種別（- = ファイル、d = ディレクトリ）
    ```
| 記号 | 意味 | 数値 |
|------|------|------|
| `r` | 読み取り（read） | 4 |
| `w` | 書き込み（write） | 2 |
| `x` | 実行（execute） | 1 |
| `-` | 権限なし | 0 |

### chmod で権限を変更する

`chmod` はファイル権限を変更するコマンドです。スクリプトを実行可能にする、秘密情報ファイルを他人から読めないようにする、という場面で使います。

```bash
# 実行権限を追加します
chmod +x script.sh

# 数値で指定します（755 = rwxr-xr-x）
chmod 755 script.sh

# 全員に読み書き実行を付与します
chmod 777 file.txt   # ← セキュリティ上、本番環境では使いません

# 所有者のみ読み書きにします
chmod 600 .env        # ← 秘密情報ファイルはこれが安全です
```

`777` は全員が読み書き実行できるため、学習用の一時対応以外では避けます。`.env` のような秘密情報ファイルは、所有者だけが読める `600` にするのが基本です。

### chown で所有者を変更する

`chown` は、ファイルやフォルダの所有者を変更するコマンドです。Web サーバーやデプロイ作業で、アプリがファイルを読めない・書けないときに関係します。

```bash
# ファイルの所有者を yamada に変更します
chown yamada README.md

# 再帰的に変更します（フォルダごと）
chown -R yamada:staff /var/www/html
```

## よく使うコマンド

### プロセス管理

> **情報工学メモ：プロセスとシグナルとは**  
> プログラムとは、ディスクに保存されたコードファイル（命令の塊）のことです。それに対して **プロセス** は、そのプログラムを実際にメモリに読み込んで「実行中の状態」にしたものです。各プロセスには **PID（プロセス ID）** という固有の番号が割り当てられます。**シグナル** は OS がプロセスに送る通知で、`kill 1234` が送る `SIGTERM` は「終了してください」という丁寧なお願い、`kill -9` の `SIGKILL` は OS が強制終了させるもので拒否できません。プロセスは「フォアグラウンド」（ターミナルを占有）か「バックグラウンド」（裏で動く）のどちらかで実行されます。

プロセス管理では、いま何が動いているかを確認し、必要なら終了させます。サーバーが重い、開発サーバーが止まらない、同じポートが使えない、といった場面で使います。

```bash
# 動いているプロセスを確認します
ps aux

# 特定のプロセスを検索します
ps aux | grep node

# リアルタイムでプロセス・CPU・メモリを確認します
top
# （終了は q キー）

# プロセスを強制終了します
kill 1234         # プロセス ID を指定します
kill -9 1234      # 強制終了します（-9 = SIGKILL）

# プロセス名で終了します
pkill node
```

`ps aux | grep node` は、動いているプロセス一覧から `node` を含む行だけを探します。`kill -9` は強制終了なので、通常はまず `kill` を使い、それでも止まらないときの最後の手段にします。

### ディスク・メモリの確認

サーバーの不調は、ディスク容量不足やメモリ不足から起きることがあります。容量やメモリの確認コマンドを知っていると、原因切り分けが速くなります。

```bash
# ディスク使用量を確認します
df -h

# フォルダのサイズを確認します
du -sh /var/log

# メモリ使用量を確認します
free -h
```

`-h` は human readable の意味で、KB や GB など人間が読みやすい単位で表示します。ログファイルが肥大化してディスクを圧迫することはよくあります。

### ファイル操作

Linux では、ファイル検索、文字列検索、ログ監視、コピー・移動・削除をコマンドで行います。特にログ調査では `grep` と `tail -f` がよく登場します。

```bash
# ファイルを検索します
find /var/log -name "*.log"

# ファイルの中身を検索します
grep "error" /var/log/nginx/error.log
grep -r "TODO" ./src/     # 再帰的に検索します

# ファイルの末尾をリアルタイム表示します（ログ監視）
tail -f /var/log/nginx/access.log

# ファイルのコピー・移動・削除
cp source.txt dest.txt
mv old.txt new.txt
rm unwanted.txt
rm -rf directory/   # フォルダごと削除します（注意：元に戻せません）
```

`rm -rf` は非常に強い削除コマンドで、元に戻せません。実行前に自分がどのディレクトリにいるか、対象パスが正しいかを必ず確認してください。

### パッケージ管理（Ubuntu/Debian）

サーバーにソフトウェアを追加するときは、OS のパッケージ管理ツールを使います。Ubuntu/Debian 系では `apt` が使われます。

```bash
sudo apt update          # パッケージリストを更新します
sudo apt install nginx   # nginx をインストールします
sudo apt remove nginx    # アンインストールします
```

`apt update` はパッケージ一覧を更新するだけで、ソフトウェア本体を更新するわけではありません。`install` や `remove` で実際に追加・削除します。

## SSH（リモートサーバーへの接続）

> **情報工学メモ：SSH の暗号化の仕組み（公開鍵・秘密鍵）**  
> SSH は通信を暗号化してリモートサーバーを安全に操作するプロトコルです。鍵認証では「公開鍵」と「秘密鍵」のペアを使います。公開鍵は南京錠、秘密鍵はその鍵と考えてください。サーバーに公開鍵（南京錠）を預けておくと、対応する秘密鍵を持つ人だけが解錠してログインできます。公開鍵は誰に見られても構いませんが、秘密鍵は絶対に他人に渡してはいけません。パスワード認証と違い、通信を傍受されても秘密鍵がなければ侵入できないため、より安全です。

**SSH**（Secure Shell）を使って、ネットワーク越しにサーバーを操作します。

SSH は、手元のターミナルからリモートサーバーへ安全にログインする仕組みです。クラウドサーバー、研究室サーバー、GPU サーバーを操作するときによく使います。

```bash
# 基本的な接続（パスワード認証）
ssh username@192.168.1.100

# 鍵認証で接続します
ssh -i ~/.ssh/id_rsa username@192.168.1.100

# 接続を切断します
exit
```

`username@192.168.1.100` は「その IP アドレスのサーバーに、username というユーザーで接続する」という意味です。鍵認証では、`-i` で使う秘密鍵ファイルを指定します。

### SSH 鍵の生成

パスワードより安全な鍵認証を使うには鍵ペアを生成します。

次のコマンドは、Ed25519 という方式の鍵ペアを作成します。公開鍵はサーバーに登録し、秘密鍵は手元で厳重に管理します。

```bash
ssh-keygen -t ed25519 -C "your@email.com"
# → ~/.ssh/id_ed25519（秘密鍵）と ~/.ssh/id_ed25519.pub（公開鍵）が生成されます
```

公開鍵（`.pub`）をサーバーに登録し、秘密鍵は絶対に他人に渡さないでください。

## シェルスクリプト

**シェルスクリプト** とは、ターミナルのコマンドをファイルにまとめて自動実行できるようにしたものです。複数のコマンドを順番に実行したり、条件分岐させたりできます。

次の例は、日付入りのバックアップファイルを作るシェルスクリプトです。1 行ずつターミナルに打っていたコマンドを、ファイルにまとめて再利用できるようにしています。

```bash
#!/bin/bash
# ↑ 「これは bash スクリプト」と宣言する 1 行目（shebang）

# 変数
DATE=$(date +%Y%m%d)   # 今日の日付（例：20240301）
BACKUP_DIR="/var/backups"

# echo でメッセージを表示します
echo "バックアップを開始します: $DATE"

# 条件分岐
if [ -d "$BACKUP_DIR" ]; then
    echo "バックアップフォルダが存在します"
else
    mkdir -p "$BACKUP_DIR"
    echo "バックアップフォルダを作成しました"
fi

# 実際のバックアップ処理
tar -czf "$BACKUP_DIR/backup_$DATE.tar.gz" /var/www/html
echo "完了しました"
```

`#!/bin/bash` は、このファイルを bash で実行するという指定です。`$DATE` や `$BACKUP_DIR` は変数の値を展開しています。`if [ -d "$BACKUP_DIR" ]` は、バックアップ用フォルダが存在するかを調べています。

### スクリプトを実行する

シェルスクリプトは、ファイルを作っただけでは実行できないことがあります。実行権限を付けてから、パスを指定して実行します。

```bash
# 実行権限を付与します
chmod +x backup.sh

# 実行します
./backup.sh
```

`./backup.sh` の `./` は、今いるディレクトリにある `backup.sh` を実行するという意味です。

## cron（定期実行）

> **情報工学メモ：デーモン（常駐プロセス）とは何か**  
> **デーモン（daemon）** とは、バックグラウンドで常に起動し続けて特定の役割を担うプロセスのことです。Web サーバー（nginx）・データベース（PostgreSQL）・cron などがこれにあたります。ユーザーがログインしていなくても動き続け、特定のイベント（時刻の到来・ネットワーク接続など）に反応して処理を行います。名前の末尾に `d` がつくことが多く（例：`sshd`・`crond`）、これは daemon の略です。

**cron** は、指定した時刻にコマンドやスクリプトを自動実行する仕組みです。バックアップ・ログの掃除・データ取得などの自動化に使います。

### crontab の書き方

cron を使うには、`crontab` に「いつ、何を実行するか」を書きます。時刻指定は左から、分・時・日・月・曜日の順です。

```bash
# crontab を編集します（エディタが開きます）
crontab -e

# 現在の設定を確認します
crontab -l
```

!!! info ""
    # 書式：分 時 日 月 曜日 コマンド
    # *（アスタリスク）= 「毎」を意味します

    # 毎日 2:30 にバックアップします
    30 2 * * * /home/yamada/backup.sh

    # 毎時 0 分に実行します（毎時）
    0 * * * * /home/yamada/hourly.sh

    # 毎週月曜 0:00 に実行します
    0 0 * * 1 /home/yamada/weekly.sh

    # 毎月 1 日の 3:00 に実行します
    0 3 1 * * /home/yamada/monthly.sh
`*` は「毎」を意味します。たとえば `30 2 * * *` は、毎日 2 時 30 分です。cron では環境変数や作業ディレクトリが普段のターミナルと違うことがあるため、スクリプト内では絶対パスを使うと安全です。

### cron のログを確認する

定期実行が動かないときは、まず cron のログを確認します。実行されたか、エラーが出ていないかを見るためです。

```bash
# Ubuntu の場合
grep CRON /var/log/syslog | tail -20
```

## Docker との関係

Docker コンテナの中身は Linux です。コンテナ内でシェルを使うと Linux コマンドが使えます。

Docker を使っていると、手元の OS が Windows や macOS でも、コンテナの中では Linux コマンドを使います。開発中に「コンテナ内で何が起きているか」を調べるために重要です。

```bash
# コンテナ内のシェルに入ります
docker compose exec web sh

# コンテナ内でコマンドを実行します
ls -la          # ファイル一覧
cat /etc/os-release  # OS 情報を確認します
```

`docker compose exec web sh` は、`web` サービスのコンテナ内で `sh` を起動するコマンドです。中に入ったら、`ls` や `cat` でファイルを確認できます。

## よくある疑問

**Q. `sudo` って何？**  
A. **S**uper **U**ser **DO** の略で、管理者権限でコマンドを実行します。インストールやシステム設定の変更に必要です。むやみに使うと危険な操作もできてしまうため注意してください。

**Q. `Permission denied` と表示される**  
A. 権限がないファイル・フォルダを操作しようとしています。`chmod` で権限を確認・変更するか、`sudo` を付けて実行してください。

**Q. `rm -rf /` は絶対にダメ？**  
A. はい。ルートディレクトリ以下の全ファイルを削除するコマンドで、OS が壊れます。スクリプト内で変数が空の場合などに意図せず実行されることがあるため、`rm -rf` を使うときは対象パスを必ず確認してください。
