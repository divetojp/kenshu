# デザインパターン

よく起きる設計上の問題に対する、再利用可能な解決策のカタログです。1994年に GoF（Gang of Four）が整理した 23 パターンが有名です。パターンの名前を知っていると「ここは Strategy パターンで解決できる」のようにチームで共通言語として使えます。

---

## はじめて読む人へ

デザインパターンは「コードのテンプレート」ではなく「設計上の考え方の名前」です。全 23 個を暗記する必要はありません。「こういう問題ならこう解決する」という引き出しとして知っておくと、コードレビューや設計議論で役立ちます。

### 読む前に押さえること

- [オブジェクト指向](オブジェクト指向.md) のクラス・継承・インターフェースの基本
- [ソフトウェア設計](ソフトウェア設計.md) の依存関係・責務分離の考え方

### 読み終えたら説明できること

- 生成・構造・行動の 3 カテゴリの違いを説明できる
- Factory・Observer・Strategy・Repository を具体例付きで説明できる
- 「なぜそのパターンを使うか」を問題の観点から説明できる

---

## 3 つのカテゴリ

| カテゴリ | 問いかけ | 代表パターン |
|---------|---------|------------|
| **生成パターン** | オブジェクトをどう作るか | Factory Method, Singleton, Builder |
| **構造パターン** | クラスをどう組み合わせるか | Adapter, Decorator, Repository |
| **行動パターン** | オブジェクト間の通信をどう設計するか | Observer, Strategy, Command |

## 生成パターン

### Factory Method — 「何を作るかを外から差し込む」

**問題：** 種類ごとにオブジェクトの作り方が違うが、呼び出し側は種類を知らなくていい。

```python
from abc import ABC, abstractmethod

class Notifier(ABC):
    @abstractmethod
    def send(self, message: str) -> None: ...

class EmailNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"メール送信: {message}")

class SlackNotifier(Notifier):
    def send(self, message: str) -> None:
        print(f"Slack送信: {message}")

# Factory: 種類を文字列で受け取り、適切なオブジェクトを返す
def create_notifier(type: str) -> Notifier:
    if type == "email":
        return EmailNotifier()
    if type == "slack":
        return SlackNotifier()
    raise ValueError(f"Unknown notifier: {type}")

# 呼び出し側は具体的なクラスを知らなくていい
notifier = create_notifier("slack")
notifier.send("デプロイ完了")
```

**使いどころ：** 設定ファイルや環境変数で切り替えたいとき。テスト時にモックに差し替えたいとき。

---

### Singleton — 「インスタンスを 1 つだけ保証する」

**問題：** DB 接続・ログ設定など、アプリ全体で 1 つだけ存在すべきオブジェクトがある。

```python
class DatabaseConnection:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._connect()
        return cls._instance

    def _connect(self):
        print("DB接続を確立（1回だけ実行）")
        self.connection = "connected"

    def query(self, sql: str):
        print(f"実行: {sql}")

# 何回 new しても同じインスタンスが返る
db1 = DatabaseConnection()
db2 = DatabaseConnection()
print(db1 is db2)  # True
```

**注意：** Singleton はグローバル状態を作るため、テストが難しくなります。モジュールレベルの変数や DI（依存性注入）で代替できる場合はそちらを検討してください。

### Builder — 「複雑なオブジェクトを段階的に組み立てる」

**問題：** コンストラクタに引数が多すぎて使いにくい。省略可能な設定が多い。

```python
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class QueryConfig:
    table: str
    conditions: list[str] = field(default_factory=list)
    limit: Optional[int] = None
    order_by: Optional[str] = None
    offset: int = 0

class QueryBuilder:
    def __init__(self, table: str):
        self._config = QueryConfig(table=table)

    def where(self, condition: str) -> "QueryBuilder":
        self._config.conditions.append(condition)
        return self  # メソッドチェーンのために自身を返す

    def limit(self, n: int) -> "QueryBuilder":
        self._config.limit = n
        return self

    def order_by(self, column: str) -> "QueryBuilder":
        self._config.order_by = column
        return self

    def build(self) -> str:
        sql = f"SELECT * FROM {self._config.table}"
        if self._config.conditions:
            sql += " WHERE " + " AND ".join(self._config.conditions)
        if self._config.order_by:
            sql += f" ORDER BY {self._config.order_by}"
        if self._config.limit:
            sql += f" LIMIT {self._config.limit}"
        return sql

# メソッドチェーンで読みやすく組み立て
sql = (
    QueryBuilder("users")
    .where("age > 20")
    .where("active = TRUE")
    .order_by("created_at DESC")
    .limit(10)
    .build()
)
print(sql)
# SELECT * FROM users WHERE age > 20 AND active = TRUE ORDER BY created_at DESC LIMIT 10
```

## 構造パターン

### Adapter — 「インターフェースの違いを吸収する」

**問題：** 既存のクラスをそのまま使いたいが、期待されるインターフェースと合わない。

```python
# 既存の外部ライブラリ（変更できない）
class LegacyPaymentAPI:
    def make_payment(self, amount_yen: int, card_num: str) -> bool:
        print(f"旧APIで決済: {amount_yen}円 / カード: {card_num}")
        return True

# 新しいシステムが期待するインターフェース
class PaymentProcessor(ABC):
    @abstractmethod
    def process(self, amount: float, card: str) -> bool: ...

# Adapter: 新インターフェースを旧APIに橋渡し
class LegacyPaymentAdapter(PaymentProcessor):
    def __init__(self, legacy: LegacyPaymentAPI):
        self._legacy = legacy

    def process(self, amount: float, card: str) -> bool:
        return self._legacy.make_payment(int(amount), card)

# 新しいコードは LegacyPaymentAPI の存在を知らなくていい
processor = LegacyPaymentAdapter(LegacyPaymentAPI())
processor.process(1500.0, "1234-5678")
```

### Decorator — 「既存の振る舞いに機能を追加する」

**問題：** 元のクラスを変更せずに、ログ記録・キャッシュ・認証などを追加したい。

```python
import time
from functools import wraps

# 関数デコレータ（Python らしい実装）
def timer(func):
    @wraps(func)
    def wrapper(*args, **kwargs):
        start = time.time()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {(time.time() - start) * 1000:.1f}ms")
        return result
    return wrapper

def cache(func):
    memo = {}
    @wraps(func)
    def wrapper(*args):
        if args not in memo:
            memo[args] = func(*args)
        return memo[args]
    return wrapper

# 複数のデコレータを重ねられる
@timer
@cache
def fibonacci(n: int) -> int:
    if n <= 1:
        return n
    return fibonacci(n - 1) + fibonacci(n - 2)

print(fibonacci(35))
```

### Repository — 「データアクセスをビジネスロジックから分離する」

Web 開発で最も重要なパターンの 1 つです。DB の操作をインターフェースの後ろに隠し、ビジネスロジックを DB から独立させます。

```python
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class User:
    id: int
    name: str
    email: str

class UserRepository(ABC):
    @abstractmethod
    def find_by_id(self, user_id: int) -> User | None: ...

    @abstractmethod
    def save(self, user: User) -> User: ...

# 本番: DB にアクセス
class PostgresUserRepository(UserRepository):
    def find_by_id(self, user_id: int) -> User | None:
        # DB クエリ（実際は SQLAlchemy 等を使う）
        return User(id=user_id, name="田中", email="tanaka@example.com")

    def save(self, user: User) -> User:
        # INSERT / UPDATE
        return user

# テスト用: メモリ内に保存（DB 不要）
class InMemoryUserRepository(UserRepository):
    def __init__(self):
        self._store: dict[int, User] = {}

    def find_by_id(self, user_id: int) -> User | None:
        return self._store.get(user_id)

    def save(self, user: User) -> User:
        self._store[user.id] = user
        return user

# ビジネスロジックは Repository のインターフェースだけに依存
class UserService:
    def __init__(self, repo: UserRepository):
        self._repo = repo

    def get_user_name(self, user_id: int) -> str:
        user = self._repo.find_by_id(user_id)
        if not user:
            raise ValueError(f"User {user_id} not found")
        return user.name

# テスト時はメモリ版を使う
service = UserService(InMemoryUserRepository())
```

## 行動パターン

### Observer — 「イベントを複数のリスナーに通知する」

**問題：** あるオブジェクトの状態変化を、複数のオブジェクトに知らせたい。

```python
from typing import Callable

class EventEmitter:
    def __init__(self):
        self._listeners: dict[str, list[Callable]] = {}

    def on(self, event: str, callback: Callable) -> None:
        self._listeners.setdefault(event, []).append(callback)

    def emit(self, event: str, *args) -> None:
        for cb in self._listeners.get(event, []):
            cb(*args)

# ユーザー登録イベントを例に
emitter = EventEmitter()

# リスナーを登録（複数 OK）
emitter.on("user_registered", lambda u: print(f"ウェルカムメール送信: {u['email']}"))
emitter.on("user_registered", lambda u: print(f"分析イベント記録: {u['id']}"))
emitter.on("user_registered", lambda u: print(f"Slack通知: {u['name']}が登録"))

# イベントを発火 → 全リスナーに通知
emitter.emit("user_registered", {"id": 1, "name": "田中", "email": "tanaka@example.com"})
```

**JavaScript/TypeScript の EventEmitter・React の useState/Context も Observer の一種**です。

### Strategy — 「アルゴリズムを差し替え可能にする」

**問題：** 同じ処理に複数のアルゴリズムがあり、状況に応じて切り替えたい。

```python
from abc import ABC, abstractmethod

class SortStrategy(ABC):
    @abstractmethod
    def sort(self, data: list) -> list: ...

class QuickSort(SortStrategy):
    def sort(self, data: list) -> list:
        if len(data) <= 1:
            return data
        pivot = data[len(data) // 2]
        return (
            self.sort([x for x in data if x < pivot]) +
            [x for x in data if x == pivot] +
            self.sort([x for x in data if x > pivot])
        )

class BubbleSort(SortStrategy):
    def sort(self, data: list) -> list:
        data = data.copy()
        for i in range(len(data)):
            for j in range(len(data) - i - 1):
                if data[j] > data[j + 1]:
                    data[j], data[j + 1] = data[j + 1], data[j]
        return data

class Sorter:
    def __init__(self, strategy: SortStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: SortStrategy) -> None:
        self._strategy = strategy  # 実行時に切り替え可能

    def sort(self, data: list) -> list:
        return self._strategy.sort(data)

sorter = Sorter(QuickSort())
print(sorter.sort([3, 1, 4, 1, 5]))

sorter.set_strategy(BubbleSort())   # 切り替え
print(sorter.sort([3, 1, 4, 1, 5]))
```

**機械学習での応用：** sklearn の `model.fit()` が Strategy パターンです。`RandomForestClassifier` や `XGBClassifier` を差し替えながら同じコードで評価できます。

## よく使われるパターン対応表

| 場面 | パターン |
|------|---------|
| 環境・設定でクラスを切り替えたい | Factory Method |
| 複数のステップで複雑なオブジェクトを作りたい | Builder |
| 古いライブラリを新しい API に合わせたい | Adapter |
| ログ・キャッシュ・認証を元のクラスを変えずに追加 | Decorator |
| DB アクセスをビジネスロジックから分離したい | Repository |
| イベント駆動（登録・通知・ログ）| Observer |
| 複数アルゴリズムを切り替えたい | Strategy |
