# ソフトウェア設計

> 「動くコード」から「長く維持できるコード」へ。設計の目的は変更コストを下げることです。[オブジェクト指向](オブジェクト指向) でクラス・継承・ポリモーフィズムを学んだあとに読むと理解が深まります。

---

## はじめて読む人へ

ソフトウェア設計は、コードを長く保守できる形に整理する考え方です。動くだけのコードから、変更しやすく、テストしやすく、読みやすいコードへ進めるために学びます。


### 読む前に押さえること

- 責務分離は、1つの部品に多くの役割を持たせすぎない考え方です。
- 依存性の注入は、外部の部品を差し替えやすくする方法です。
- リファクタリングは、動作を変えずに内部構造を改善する作業です。

### 読み終えたら説明できること

- 責務分離、依存性注入、リファクタリングの目的を説明できる。
- テストしにくい設計の特徴を理解できる。
- 保守性のための基本原則を説明できる。

## 責務分離（Single Responsibility Principle）

責務分離とは、1つの関数やクラスに多くの役割を持たせすぎない考え方です。データを取得する、加工する、保存する、表示する、という処理が1か所に混ざると変更しにくくなります。

責務が分かれていると、変更の影響範囲を小さくできます。保存方法を変えたいときに表示ロジックまで触らなくてよい、という状態が保守しやすい設計です。

1 つのクラス・関数は 1 つの責務だけを持ちます。

### 責務が混在した例

次の関数は、注文データの検証、DB保存、メール送信、ログ出力をすべて1か所で行っています。動くコードとしては成立しますが、「注文処理」という名前の中に複数の理由で変更される処理が混ざっています。

```python
def handle_order(order_data: dict):
    # バリデーション
    if not order_data.get("user_id"):
        raise ValueError("user_id is required")

    # DB 保存
    db.execute("INSERT INTO orders ...", order_data)

    # メール送信
    send_email(order_data["email"], "ご注文ありがとうございます")

    # ログ出力
    logger.info(f"Order created: {order_data}")
```

この関数が変わる理由が 4 つあります（バリデーションルール変更・DB スキーマ変更・メール内容変更・ログ形式変更）。どれか 1 つを変えるたびに 4 つすべての動作を確認しなければなりません。

### 責務を分けた例

責務を分けると、`handle_order` は全体の流れを調整するだけになります。細かいルールは `validate_order`、保存は `save_order`、通知は `notify_user` に分かれます。

```python
def validate_order(data: dict) -> None:
    if not data.get("user_id"):
        raise ValueError("user_id is required")

def save_order(data: dict) -> int:
    return db.execute("INSERT INTO orders ...", data)

def notify_user(email: str) -> None:
    send_email(email, "ご注文ありがとうございます")

def handle_order(order_data: dict):
    validate_order(order_data)
    order_id = save_order(order_data)
    notify_user(order_data["email"])
    logger.info(f"Order created: {order_id}")
```

各関数は独立してテストでき、変更の影響範囲が明確になります。

たとえばメール文面を変えるときは `notify_user` を見ればよく、DBの保存形式を変えるときは `save_order` を見ればよくなります。設計の目的は、コードを細かく分けること自体ではなく、「変更したい理由ごとに触る場所を分ける」ことです。

## 依存関係の管理

### 依存関係の方向性

「上位モジュールが下位モジュールに依存する」のが自然な構造です。問題になるのは循環依存（A → B → A）です。

!!! info ""
    ```
    良い例：
    views → services → repositories → DB
    
    悪い例（循環依存）：
    module_a → module_b → module_a  ← どちらかを変えると両方壊れます
    ```

依存関係の向きは、変更の影響がどちらへ広がるかを決めます。`views → services → repositories → DB` のように一方向なら、画面側の変更がDB層へ不要に漏れにくくなります。循環依存があると、片方を読み込むためにもう片方が必要になり、テストや初期化の順番も複雑になります。

### 依存性の注入（Dependency Injection）

依存するオブジェクトを内部で作らず、外から渡します。

「内部で作る」と、そのクラスは特定の実装から離れられなくなります。`OrderService` が自分で `PostgresDB()` や `SMTPMailer()` を作ると、テストでも本物のDBやメール送信に引きずられます。外から渡す形にすると、本番用とテスト用を差し替えられます。

```python
# 悪い例：内部で依存を作る
class OrderService:
    def __init__(self):
        self.db = PostgresDB()      # 直接依存
        self.mailer = SMTPMailer()  # 直接依存

# 良い例：外から渡す（DI）
class OrderService:
    def __init__(self, db, mailer):
        self.db = db
        self.mailer = mailer

# テストでは Mock を渡せます
service = OrderService(db=MockDB(), mailer=MockMailer())
```

DI により、テスト時に実際の DB やメールサーバーを使わずにテストできます。

この設計では、`OrderService` は「DBに保存する」「メールを送る」という役割を使いますが、具体的にPostgreSQLなのか、モックなのか、SMTPなのかは知りません。呼び出し側が必要な部品を渡すことで、サービス本体のロジックを外部環境から切り離せます。

> **モック（Mock）とは？** テスト時に実際の DB・メール・外部 API の代わりに使う「偽物オブジェクト」です。決まった値を返すだけなので、テストが速く・安定します。

## テスト

### テストの種類

| 種類 | 範囲 | 速度 | 用途 |
|------|------|------|------|
| 単体テスト（Unit） | 関数・クラス 1 つ | 速い | ロジックの検証 |
| 結合テスト（Integration） | 複数コンポーネントの連携 | 中程度 | DB・外部 API との統合 |
| E2E テスト（End-to-End） | ユーザー操作全体 | 遅い | UI の動作確認 |

**テストピラミッド：** 単体テストを多く、E2E テストを少なく保つことで、速くて安定したテストスイートになります。

### テストの書き方（Python）

テストは、設計の良し悪しを映す鏡です。関数が小さく、外部依存が分かれていれば、テストは短く書けます。逆に、1つの関数がDB、メール、時刻、外部APIを全部触っていると、準備が大きくなり、テストが壊れやすくなります。

```python
# pytest の例
def test_validate_order_raises_when_no_user_id():
    with pytest.raises(ValueError, match="user_id is required"):
        validate_order({})

def test_validate_order_passes_with_valid_data():
    validate_order({"user_id": 1, "email": "a@example.com"})  # エラーなし
```

1つ目のテストは、`user_id` がないと `ValueError` が出ることを確認しています。2つ目のテストは、正しいデータなら例外が出ないことを確認しています。テスト名が具体的だと、失敗したときに仕様のどこが壊れたかがすぐ分かります。

**良いテストの条件：**
- テスト名が「何を・どうすると・どうなる」を表しています
- 1 テスト = 1 つのことを検証します
- 外部依存（DB・メール・時刻）はモックで置き換えます

## リファクタリング

外部からの動作を変えずに、コードの内部構造を改善することです。  
**「まず動かす → テストで保護 → リファクタ」の順番が鉄則です。テストなしのリファクタは危険です。**

### よくあるリファクタリングパターン

**マジックナンバーを定数にする：**
```python
# Before
if age < 18:
    ...

# After
ADULT_AGE = 18
if age < ADULT_AGE:
    ...
```

`18` のような数値が直接書かれていると、後から読んだ人は「これは成人年齢なのか、サービス利用可能年齢なのか、別の制限なのか」を推測しなければなりません。`ADULT_AGE` のように名前をつけると、値の意味がコードに残ります。

**長い関数を分割する：**  
関数が 20〜30 行を超えてきたら、責務を切り出せないか考えます。

**早期リターンで深いネストを解消：**
早期リターンは、「処理を続けられない条件」を先に返してしまう書き方です。正常な処理が右へ右へと深くネストしないため、主な流れを読みやすくできます。

```python
# Before（ネストが深い）
def process(data):
    if data:
        if data.get("user_id"):
            if data.get("amount") > 0:
                do_something(data)

# After（早期リターン）
def process(data):
    if not data:
        return
    if not data.get("user_id"):
        return
    if data.get("amount") <= 0:
        return
    do_something(data)
```

Before では、実際にやりたい `do_something(data)` に到達するまで3段階の `if` を読み下る必要があります。After では、条件を満たさないケースを先に返し、最後に正常系だけが残ります。エラーや例外的なケースを入口で処理することで、中心となる処理が見えやすくなります。

## 保守性のための原則

### YAGNI（You Ain't Gonna Need It）

「今必要でない機能を今作らない」という原則です。将来を想定した汎用化は、実際に必要になってから行います。

### DRY（Don't Repeat Yourself）

同じロジックを複数箇所に書きません。変更時に 1 か所だけ修正すれば済むように関数・定数に抽出します。ただし「よく似た」コードを無理やり共通化するのも問題です（早すぎる抽象化）。

**3 回書いたら共通化を検討します（Rule of Three）。**

### コードレビューで見るポイント

- 関数の責務が明確かどうか
- 変数・関数の名前が意味を表しているかどうか
- エラーハンドリングが漏れていないかどうか
- テストがあるかどうか
- セキュリティ上の問題がないかどうか（SQL インジェクション・ハードコードされた秘密情報）
