データパイプライン¶

データの取得・前処理・特徴量生成・モデル学習・評価を自動化した一連の処理フロー です。Jupyter Notebook での手作業を「再現可能な自動処理」に変えます。

データサイエンスの学習では、最初は Notebook 上で試行錯誤するのが自然です。しかし、分析結果を他の人に再現してもらう、毎日新しいデータでモデルを更新する、本番環境で同じ前処理を使う、という段階になると、手作業のままでは限界があります。

データパイプラインの目的は、誰が・いつ・どの環境で実行しても同じ手順で処理できる状態 を作ることです。これは単なる自動化ではなく、分析を信頼できるものにするための土台です。

この章で学ぶこと¶

  • Notebook での試行錯誤と、本番向けパイプラインの違いを理解する
  • sklearn Pipeline で前処理とモデルを一体化する
  • Makefile でデータ準備・学習・評価の手順をコマンド化する
  • DVC でデータやモデルのバージョンを管理する
  • Airflow のようなワークフロー管理ツールが必要になる場面を理解する

はじめて読む人へ¶

データパイプラインは、データ取得から前処理、学習、評価までを再現可能にする流れです。手作業のNotebookを、誰でも同じ順序で実行できる処理へ整えていきます。

読む前に押さえること¶

  • 各ステップの入力と出力を明確にします。
  • Pipeline化すると、学習時と推論時の前処理のズレを防げます。
  • DVC や Airflow は、データや実行順序を管理するために使います。

読み終えたら説明できること¶

  • Notebook とパイプラインの違いを説明できる。
  • sklearn Pipeline がデータリークを防ぐ理由を理解できる。
  • Makefile、DVC、Airflow の役割を区別できる。

パイプラインの全体像¶

機械学習プロジェクトの処理は、だいたい次の流れになります。

パイプラインを考えるときは、「処理の名前」だけでなく「各処理が何を受け取り、何を出すか」を一緒に考えます。入力と出力が曖昧なままだと、後から自動化しようとしても、どこから再実行すべきか分からなくなります。

  1. データ取得
  2. 前処理
  3. 特徴量生成
  4. 学習
  5. 評価
  6. 保存・デプロイ

この図は、上から下へ一方向にデータが加工されていく流れを表しています。各ステップは前のステップの出力を受け取り、次のステップの入力になる成果物を作ります。

Notebook ではこの流れを 1 つのファイルに書きがちです。探索段階ではそれで問題ありませんが、セルの実行順序が変わったり、途中の変数が残ったりすると、同じ結果を再現できなくなります。

パイプライン化すると、各ステップの入力と出力がはっきりします。たとえば「raw.csv から processed.csv を作る」「processed.csv から pipeline.pkl を作る」のように、処理の依存関係を明示できます。

Notebook からパイプラインへ移すタイミング¶

すべての分析を最初から厳密なパイプラインにする必要はありません。1 年生の学習段階では、まず Notebook で試し、うまくいった処理を少しずつ .py ファイルやパイプラインに移すのが現実的です。

段階 主な目的 使う道具
探索 データを理解する Jupyter Notebook, pandas, 可視化
整理 同じ処理を再実行できるようにする Python スクリプト, sklearn Pipeline
自動化 コマンド 1 つで一連の処理を動かす Makefile, DVC
運用 定期実行・監視・失敗時の再実行を行う Airflow, Prefect, Dagster

「この処理をもう一度やるには、どのセルをどの順番で実行するんだっけ」と迷い始めたら、パイプライン化の合図です。

sklearn Pipeline¶

sklearn Pipeline は、前処理とモデルを1本の流れとしてまとめる仕組みです。欠損値補完、スケーリング、エンコーディング、モデル学習を順番に実行できます。

重要なのは、学習時と推論時で同じ処理を使えることです。Notebook で前処理を手作業で行うと、推論時に同じ処理を忘れる危険があります。Pipeline にまとめることで、そのズレを防げます。

前処理とモデルを一本化します。データリークを防ぎ、推論時の手順も自動化されます。

機械学習では、学習時と推論時で同じ前処理を行う必要があります。学習時に標準化したのに、推論時に標準化を忘れると、モデルの入力分布が変わって予測が悪くなります。

sklearn の Pipeline は、前処理とモデルを 1 つのオブジェクトとして扱う仕組みです。fit() を呼ぶと前処理器とモデルが順番に学習され、predict() を呼ぶと同じ前処理を通してから予測されます。

特に重要なのが データリーク の防止です。テストデータを使って平均・標準偏差・欠損値補完の値を計算してしまうと、本来知らないはずの情報を学習に使ったことになります。Pipeline と ColumnTransformer を使うと、訓練データだけで前処理を fit し、テストデータには transform だけを適用できます。

次のコードは、数値列とカテゴリ列を別々に前処理し、その後で分類モデルへ渡す例です。ここでの中心概念は「前処理もモデルの一部として扱う」ことです。

from sklearn.pipeline import Pipeline
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import StandardScaler, OneHotEncoder
from sklearn.impute import SimpleImputer
from sklearn.ensemble import RandomForestClassifier

# 数値列・カテゴリ列の前処理を分けて定義
numeric_transformer = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("scaler", StandardScaler()),
])

categorical_transformer = Pipeline([
    ("imputer", SimpleImputer(strategy="most_frequent")),
    ("onehot", OneHotEncoder(handle_unknown="ignore")),
])

# ColumnTransformer で列ごとに前処理を割り当て
preprocessor = ColumnTransformer([
    ("num", numeric_transformer, ["age", "income", "score"]),
    ("cat", categorical_transformer, ["city", "category"]),
])

# パイプライン全体
pipeline = Pipeline([
    ("preprocessor", preprocessor),
    ("model", RandomForestClassifier(n_estimators=100, random_state=42)),
])

# 訓練・評価・推論がすべてパイプライン 1 本で完結
pipeline.fit(X_train, y_train)
print(f"精度: {pipeline.score(X_test, y_test):.3f}")
y_pred = pipeline.predict(X_new)

# 保存
import joblib
joblib.dump(pipeline, "pipeline.pkl")

このコードでは、数値列とカテゴリ列で別々の前処理を行っています。数値列は欠損値を中央値で埋めてから標準化し、カテゴリ列は欠損値を最頻値で埋めてから One-Hot Encoding します。

ColumnTransformer は「どの列にどの前処理をかけるか」を管理します。Pipeline は「前処理の後にモデルを学習する」という順序を管理します。この 2 つを組み合わせることで、前処理の漏れや順序の間違いを減らせます。

保存するときは、モデルだけでなくパイプライン全体を保存します。そうすると、推論時にも同じ前処理が自動的に使われます。

保存したパイプラインは、別の Python プログラムから読み込んで使えます。次のコードは、学習済みの前処理とモデルをまとめて復元し、新しいデータに対して予測する例です。

loaded = joblib.load("pipeline.pkl")
pred = loaded.predict(X_new)

この形にしておくと、FastAPI やバッチ処理からも同じ pipeline.pkl を読み込んで使えます。

loaded.predict(X_new) を呼ぶと、内部では「欠損値補完 → 標準化・One-Hot Encoding → モデル予測」が同じ順番で実行されます。アプリ側が前処理を覚えておく必要がないため、学習時と推論時のズレを防げます。

Makefile によるタスク管理¶

Makefile は、よく使うコマンドに名前を付けるためのファイルです。もともとは C 言語などのビルドに使われてきましたが、データ分析でも「データ準備」「学習」「評価」をまとめるのに便利です。

ポイントは、タスク同士の依存関係を書けることです。次の例では、train は data に依存しています。つまり make train を実行すると、必要なら先に make data が実行されます。

Makefile では、ターゲット名: の下に、そのターゲットを実行するためのコマンドを書きます。データ分析では、よく使う長いコマンドに短い名前を付けるだけでも十分価値があります。

# Makefile

.PHONY: data train evaluate clean

DATA_DIR = data
MODEL_DIR = models

# データの取得と前処理
data:
    python scripts/fetch_data.py --output $(DATA_DIR)/raw.csv
    python scripts/preprocess.py --input $(DATA_DIR)/raw.csv --output $(DATA_DIR)/processed.csv

# モデル訓練
train: data
    python scripts/train.py \
        --input $(DATA_DIR)/processed.csv \
        --output $(MODEL_DIR)/pipeline.pkl \
        --n_estimators 100

# 評価
evaluate: train
    python scripts/evaluate.py \
        --model $(MODEL_DIR)/pipeline.pkl \
        --test $(DATA_DIR)/test.csv

# クリーン
clean:
    rm -rf $(DATA_DIR)/processed.csv $(MODEL_DIR)/*.pkl

この例では、data、train、evaluate、clean がタスク名です。train: data は「train を実行する前に data が必要」という依存関係を表します。

Makefile のコマンド行は、通常タブでインデントします。スペースだけで書くと動かないことがあるため、エラーが出たときはインデントも確認してください。

実行するときは、ターミナルで make タスク名 と入力します。

make data     # データ準備のみ
make train    # データ準備 + 訓練
make evaluate # 全ステップ実行 + 評価

Makefile を使うと、手順がドキュメント化されます。README に「まずこのスクリプトを実行して、次にこのファイルを作って」と長く書く代わりに、make evaluate のような短いコマンドで再現できます。

チームで作業するときは、メンバーごとにコマンドの打ち方が違うと結果がずれます。Makefile に集約しておくと、全員が同じ入口から処理を実行できます。

DVC（Data Version Control）¶

モデル・データセットをバージョン管理します。Git が .pkl や .csv を管理できない問題を解決します。

Git はソースコードの履歴管理には強いですが、大きな CSV、画像データ、学習済みモデルの管理には向いていません。リポジトリが重くなり、差分も読みづらくなります。

DVC は、大容量ファイル本体を S3 や Google Drive などの外部ストレージに置き、Git には「どのファイルのどのバージョンを使うか」という小さなメタデータだけを保存します。これにより、コードとデータの対応関係を保ったまま履歴管理できます。

まず DVC をインストールし、プロジェクトに DVC の管理情報を作ります。dvc init は、そのリポジトリで DVC を使い始めるための初期化コマンドです。

pip install dvc
dvc init

# リモートストレージ（S3・Google Drive など）の設定
dvc remote add -d myremote s3://my-bucket/dvc

dvc remote は、大きなファイルの本体を置く保存先です。GitHub には小さな管理ファイルだけを置き、データ本体は S3 や Google Drive などに置く、という分担になります。

次に、大きなデータファイルを DVC で追跡します。Git が直接 raw.csv を持つのではなく、DVC が作る .dvc ファイルを Git に記録する点が重要です。

# 大容量ファイルを DVC で追跡（Git には .dvc ファイルのみ記録）
dvc add data/raw.csv
git add data/raw.csv.dvc .gitignore
git commit -m "add raw dataset"
dvc push   # S3 などにアップロード

# ファイルの取得（他のマシンや CI/CD 環境で）
dvc pull

dvc push はデータ本体をリモートストレージへアップロードし、dvc pull は必要なデータ本体を手元に取得します。チームメンバーは Git でコードとメタデータを取り、DVC でデータ本体を取る、という流れになります。

DVC はファイル単体の管理だけでなく、処理手順の管理もできます。次の dvc.yaml は、前処理、学習、評価の依存関係を定義しています。

# dvc.yaml（パイプラインの定義）
stages:
  preprocess:
    cmd: python scripts/preprocess.py
    deps:
      - data/raw.csv
      - scripts/preprocess.py
    outs:
      - data/processed.csv

  train:
    cmd: python scripts/train.py
    deps:
      - data/processed.csv
      - scripts/train.py
    outs:
      - models/pipeline.pkl
    metrics:
      - metrics/scores.json

  evaluate:
    cmd: python scripts/evaluate.py
    deps:
      - models/pipeline.pkl
      - data/test.csv
    metrics:
      - metrics/test_scores.json

deps はそのステージが必要とする入力、outs はそのステージが作る出力、metrics は評価指標として比較したいファイルです。この構造にしておくと、どの入力が変わったときにどの処理をやり直すべきかを DVC が判断できます。

定義したパイプラインは、次のコマンドで実行・確認します。

dvc repro        # 変更があった段階から再実行（差分実行）
dvc dag          # パイプラインの依存関係を表示
dvc metrics show # メトリクスの比較

dvc.yaml は、処理の依存関係をファイルとして定義します。deps は入力、outs は出力、metrics は評価指標です。DVC は入力ファイルやスクリプトが変わったステップだけを再実行できるため、重い処理を毎回全部やり直さずに済みます。

たとえば前処理スクリプトだけを変更した場合、preprocess 以降の train と evaluate は再実行されます。一方、評価スクリプトだけを変更した場合は、学習済みモデルを作り直さずに評価だけ実行できます。

パイプライン設計で決めること¶

パイプラインを書く前に、次の 4 つを紙に書き出すと設計しやすくなります。

決めること 例
入力データ data/raw.csv、API、データベース
中間データ data/processed.csv、特徴量ファイル
成果物 models/pipeline.pkl、metrics/scores.json
実行単位 取得、前処理、学習、評価、デプロイ

初心者が最初から複雑なワークフロー管理ツールを使う必要はありません。まずは「各ステップの入力と出力を明確にする」だけでも、再現性は大きく上がります。

Airflow（バッチ処理の定期実行）¶

本番環境で毎日・毎週データを更新するようなバッチ処理に使います。

Airflow は、複数のタスクを決まった順番で実行するためのワークフロー管理ツールです。DAG（Directed Acyclic Graph）という形で「どのタスクがどのタスクの後に実行されるか」を表します。

Makefile や DVC は、主に手元や CI で「必要なときに実行する」ために使います。一方、Airflow は「毎日 3 時にデータを取得して、学習して、結果を保存する」のような定期実行に向いています。

次のコードは、Airflow の DAG を Python で定義する例です。DAG は「タスクの依存関係を表すグラフ」であり、処理そのものだけでなく、実行順序を明示するための設計図です。

from airflow.decorators import dag, task
from datetime import datetime, timezone

@dag(schedule_interval="@daily", start_date=datetime(2024, 1, 1))
def ml_pipeline():

    @task
    def fetch_data():
        import pandas as pd
        import requests
        data = requests.get("https://api.example.com/daily-data").json()
        df = pd.DataFrame(data)
        date = datetime.now(timezone.utc).strftime("%Y%m%d")
        path = f"/data/raw_{date}.csv"
        df.to_csv(path, index=False)
        return path

    @task
    def train_model(data_path: str):

        import pandas as pd
        df = pd.read_csv(data_path)
        # 訓練処理...
        joblib.dump(pipeline, "/models/pipeline_latest.pkl")
        return "/models/pipeline_latest.pkl"

        # モデルをサービングサーバーに配置

Airflow では、各関数が 1 つのタスクになります。path = fetch_data()、model = train_model(path)、deploy_model(model) という書き方で、タスクの順番が決まります。

実務では、タスクが失敗したときの再試行、ログの確認、通知、古いデータの扱いも重要になります。1 年生の段階では、まず「処理を小さなタスクに分け、依存関係を明示する」という考え方を押さえてください。

このサンプルは概念を理解するための最小例です。実際の本番環境では、API のエラー、認証情報、保存先の権限、再実行時に同じファイルを上書きしてよいか、といった運用上の設計も必要になります。

まとめ：ツールの使い分け¶

フェーズ ツール
実験・探索 Jupyter Notebook
再現可能な処理 sklearn Pipeline + Makefile
データバージョン管理 DVC
実験記録 MLflow / wandb
本番バッチ処理 Airflow / Prefect / Dagster
モデル提供 FastAPI + Docker

道具の名前を覚えることよりも、どの問題を解くための道具なのかを理解することが大切です。sklearn Pipeline は前処理の再現性、Makefile は実行手順の統一、DVC はデータとモデルの履歴、Airflow は定期実行を担当します。

よくある失敗¶

Notebook のセル実行順に依存する¶

上から順番に実行していない Notebook は、他の人が再現できません。重要な処理はスクリプトやパイプラインに移し、初期状態から実行しても同じ結果になるようにします。

テストデータで前処理を fit してしまう¶

標準化、欠損値補完、エンコーディングは、訓練データだけで fit します。テストデータは transform だけです。これを守らないと、評価結果が実際より良く見えることがあります。

データやモデルのバージョンが分からない¶

「どのデータで学習したモデルか」が分からないと、結果を比較できません。最低限、データファイル名、作成日時、評価指標、学習コードのコミットを記録してください。

いきなり大規模ツールを導入する¶

小さな課題なら、Makefile と sklearn Pipeline だけで十分なことも多いです。Airflow のようなツールは、定期実行や監視が必要になってから導入しても遅くありません。

確認問題¶

  1. Notebook だけで分析を進めると、再現性の面でどのような問題が起きますか。
  2. sklearn Pipeline がデータリークを防ぐ理由を説明してください。
  3. Git ではなく DVC を使うべきファイルの例を 2 つ挙げてください。
  4. Makefile と Airflow の役割の違いを説明してください。

ミニ演習¶

  1. fetch_data.py、preprocess.py、train.py、evaluate.py の 4 つの処理があると仮定し、それぞれの入力と出力を書き出してください。
  2. その処理を make evaluate で最後まで実行できる Makefile にしてください。
  3. sklearn Pipeline で「欠損値補完 → 標準化 → ロジスティック回帰」を 1 つのパイプラインとして書いてください。
  4. 作ったモデルを保存し、別ファイルから読み込んで predict() する流れを書いてください。
